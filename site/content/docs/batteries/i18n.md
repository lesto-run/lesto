# Internationalization

`@lesto/i18n` is the internationalization core: message catalogs, `{placeholder}`
interpolation, and pluralization. It's a single class — `I18n` — and it has no
dependencies. The one idea it leans on: translation must never fail loudly. A
missing key surfaces as the key itself rather than a blank, and a pluralized
count picks its grammatical category from the platform's `Intl.PluralRules`, so
the right form is chosen per language instead of hard-coded to English's
`one`/`other`.

## Define catalogs

You construct an `I18n` with a `defaultLocale` and a map of `locales`. Each
locale is a flat `Messages` catalog — a `Record<string, string>` from key to
template. Templates use `{name}` placeholders, filled from `params` at
translation time.

```ts
import { I18n } from "@lesto/i18n";

const i18n = new I18n({
  defaultLocale: "en",
  locales: {
    en: { hello: "Hello, {name}!", "cart.one": "{count} item", "cart.other": "{count} items" },
    fr: { hello: "Bonjour, {name} !" },
  },
});
```

The catalog is deliberately flat. Keys are dotted *by convention* — `cart.one`,
`cart.other` for the plural forms of `cart` — but the catalog stores them as
plain string keys, not nested objects. That keeps lookup a single own-property
read and keeps the data trivially serializable from JSON.

## Translate a key

`t(locale, key, params?)` resolves a template and interpolates it. Lookup order
is the requested locale first, then — when `fallback` is on (the default) — the
`defaultLocale`. A key that resolves in neither comes back as the key string
itself, so an untranslated string is visible on screen rather than silently
empty.

```ts
i18n.t("fr", "hello", { name: "Ada" }); // "Bonjour, Ada !"
i18n.t("fr", "missing.key");            // "missing.key" — absent from both fr and en
i18n.has("en", "hello");                // true — resolves (honouring fallback)
i18n.locales();                         // ["en", "fr"]
```

`has(locale, key)` answers whether a key resolves at all (it honours fallback
too), and `locales()` lists the locale names the instance was built with —
useful for rendering a language switcher.

Interpolation is exported on its own as `interpolate(template, params)` if you
need it without a catalog. A placeholder with no matching param is left verbatim
(`{name}` stays as written), which makes missing data obvious rather than
blank.

```ts
import { interpolate } from "@lesto/i18n";

interpolate("Hi {name}, you have {n} messages", { name: "Ada", n: 3 });
// "Hi Ada, you have 3 messages"
```

## Pluralize by count

`plural(locale, key, count, params?)` chooses a CLDR plural category for `count`
using the locale's `Intl.PluralRules`, then looks up `key.<category>`. The
`count` is threaded into the params automatically, so the chosen template can
render it as `{count}`.

```ts
i18n.plural("en", "cart", 1); // "1 item"   → cart.one
i18n.plural("en", "cart", 3); // "3 items"  → cart.other
```

Because the category comes from `Intl.PluralRules`, pluralization is
locale-correct without per-language code: `fr` treats `0` as singular, `ru` and
`pl` split `few` from `many`, and `ar` spans all six categories. A catalog only
needs to spell the categories its language actually uses — English needs `one`
and `other` — and `other` is the universal fallback, so a key spelled with just
`cart.other` keeps resolving for every count.

## Catch missing translations

Returning the raw key keeps the app running, but you still want to *know* a
string went untranslated. `onMissing` is an observability seam: it fires once
per `t` call whose key resolves nowhere, naming the originally-requested locale.
Wire it to a counter and you find untranslated strings in production before a
user reports a raw key on screen.

```ts
const i18n = new I18n({
  defaultLocale: "en",
  locales: { en, fr },
  fallback: true,
  onMissing: (locale, key) => metrics.increment("i18n.miss", { locale, key }),
});
```

The hook is purely observational — `t` still returns the key regardless of what
it does — and a throw from inside it is swallowed, so a broken counter can never
break a translation. See [Observability](/batteries/observability) for where
that counter lives.

## Notes and gotchas

- **Output is plain text — escape at the render layer.** `interpolate` splices
  param values in as-is with no HTML, attribute, or URL encoding. Both the
  template and the params are untrusted from a markup standpoint, so the caller
  MUST escape the result for its sink (HTML body vs. attribute vs. URL) before
  writing it into a document. The module does no escaping on purpose, because
  only the renderer knows the correct encoding. See
  [Validation](/guides/validation) for boundary-encoding practice.
- **A missing key is the key, not a blank.** `t` never throws and never returns
  an empty string; an unresolved key comes back verbatim. Pair that with
  `onMissing` so "visible on screen" is also "counted in metrics".
- **Fallback is on by default.** With `fallback: true`, a key absent from the
  requested locale is read from `defaultLocale`. Set `fallback: false` to make a
  locale stand alone — then an absent key resolves to the key itself directly,
  with no cross-locale read.
- **Plurals prefer the requested locale's own forms.** Resolution favours the
  requested locale's category (then its `other`) before borrowing the default
  locale, so a half-translated catalog spelled with only `key.other` stays in
  its own language rather than bleeding the default locale's exact-category
  string. Only when the requested locale spells neither form does it defer to
  the default locale, re-selecting the category by that locale's rules so the
  borrowed plural is itself grammatical.
- **An unparseable locale tag degrades to `other`, it doesn't throw.** A
  structurally invalid BCP-47 tag that `Intl.PluralRules` rejects is recorded
  once and pluralization collapses to the `other` category, keeping `plural`
  total like `t`.
- **Lookups are own-property only.** Both catalog and param reads use
  `Object.hasOwn`, so a key like `"constructor"` or `"toString"` resolves to a
  miss instead of leaking an inherited `Object.prototype` member.
- **This is the framework-agnostic core.** `@lesto/i18n` is a pure class with no
  routing or request integration baked in. Pick the locale however your app does
  — an `Accept-Language` header, a path prefix, a session — and pass it to `t`
  and `plural`. See [Routing & pages](/guides/routing) for reading request data,
  and [Quickstart](/quickstart) and [Concepts](/concepts) for where batteries
  fit overall.
