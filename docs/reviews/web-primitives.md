# Comms, SEO & Web Primitives — Architecture Review

Reviewed: 2026-06-11 · Packages: `@volo/mail`, `@volo/mailing-lists`, `@volo/feeds`, `@volo/seo`, `@volo/i18n` · Reviewer: web-primitives domain pass (source + tests read in full).

## Domain summary

The three pure packages (feeds, seo, i18n) are small, genuinely built, well-escaped string builders with real tests — they are launch-shaped, with only spec-validity and locale-correctness gaps. The two stateful packages are honest skeletons of their pitch: `@volo/mail` is a clean queue-backed mailer with **no transport implementation anywhere in the repo** — only in-test in-memory transports exist, so Volo cannot deliver a single real email today — and `@volo/mailing-lists` implements half the double-opt-in dance (it mints the token but never sends the confirmation email), with a broadcast loop that is not crash-safe and an email schema that cannot carry the `List-Unsubscribe` header modern bulk delivery requires. Verdict: SEO/feeds/i18n can ship with P2 cleanups; the comms stack has one P0 (transport) and a cluster of P1s before "batteries-included email" is a true claim.

## Performance

- **P2** — `packages/mailing-lists/src/models.ts:147-159` — the migration creates `lists` and `subscribers` with no indexes; `confirm`/`unsubscribe` look up by token (`models.ts:97-114`) and `broadcast` filters by `(list_id, status)` (`models.ts:130-136`), so every call full-scans `subscribers`. **Recommend:** add a `UNIQUE` index on `subscribers.token` (also fixes token-collision flipping the wrong row) and a composite index on `(list_id, status)` in the migration.
- **P2** — `packages/mailing-lists/src/mailing-lists.ts:117-125` — `broadcast` loads every subscribed row into memory (`.all()`) then awaits one `enqueue` INSERT per recipient, serially. At 100k subscribers that is 100k sequential round trips inside a single call, and each job row duplicates the full `params` payload. **Recommend:** chunked batch enqueue inside a transaction (the queue and db already share one `SqlDatabase` seam, as the package's own test proves at `test/mailing-lists.test.ts:101-104`).
- Healthy: feeds/seo/i18n are allocation-light pure functions (fixed `replaceAll` pipelines, one regex pass per interpolation — `packages/seo/src/escape.ts:15-22`, `packages/i18n/src/interpolate.ts:13-23`); nothing to do before 1.0.

## Security

- **P1** — `packages/mail/src/mailer.ts:108-113` — email header injection: `deliver` hands `to`, `subject`, `from` to the transport with no CRLF validation anywhere in the package. Today this is latent only because no real transport exists; the day an SMTP/provider transport lands, `subscribe(listId, "a@b.com\r\nBcc: victims...")` (mailing-lists stores arbitrary strings, see next finding) flows straight into headers. The Mailer is the contract point. **Recommend:** reject `\r`/`\n` (and bare `%0d%0a` after decode at the HTTP boundary) in `to`/`subject`/`from` with a coded `MAIL_INVALID_ADDRESS`/`MAIL_INVALID_HEADER` error at both `define`-build time and `deliver`.
- **P1** — `packages/mailing-lists/src/mailing-lists.ts:87-94` — `subscribe` accepts any string as `email` with no syntax validation, no per-list uniqueness (the schema at `models.ts:37-45` has no unique constraint), and no rate limiting. Consequences: duplicate rows → duplicate broadcast sends to the same address; subscription bombing of a victim address; CRLF-bearing "emails" persisted and later replayed as `to` (feeds the P1 above). **Recommend:** validate shape, upsert on `(list_id, email)` with a unique index, and document that the HTTP boundary must rate-limit (per ADR 0005, validation lives at the boundary — but uniqueness is storage's job).
- **P2** — `packages/mailing-lists/src/models.ts:97-102` + `mailing-lists.ts:106-114` — the confirmation token doubles as a permanent unsubscribe token: same column, never rotated after `confirm`. Entropy is good (128-bit `randomBytes(16)`, `mailing-lists.ts:38`) and the indexed-equality lookup is standard, but a forwarded confirmation email leaks a forever-valid unsubscribe (and re-subscribe-state-tamper) capability. **Recommend:** rotate the token on `confirm`, or keep distinct `confirmToken`/`unsubscribeToken` columns.
- **P2** — `packages/seo/src/robots.ts:18-28` — `Allow`/`Disallow` paths and the `sitemap` URL are interpolated with no newline stripping; a `\n`-bearing path injects arbitrary robots directives (e.g. flipping `Disallow: /admin` to allowed). Inputs are usually developer-supplied, hence P2 not P1. **Recommend:** strip/reject `\r\n` and `#` in paths.
- **P2** — `packages/i18n/src/interpolate.ts:13-23` — params are interpolated as raw text with no HTML escaping; this is correct for a transport-agnostic core, but nothing states the contract, and `t()` output will routinely be dropped into HTML. **Recommend:** document "output is plain text; escape at the render layer" in the module doc, and have `@volo/ui` treat translations as text nodes by default.
- Verified solid: feeds escapes all five XML entities ampersand-first in both element text and attributes (`packages/feeds/src/xml.ts:17-22`, exercised at `rss.ts`/`atom.ts` every interpolation site); seo uses the same pipeline (`packages/seo/src/escape.ts:15-22`) on every meta/sitemap value; `jsonLd` neutralizes `</script>` breakout via `<` → `<` (`packages/seo/src/json-ld.ts:17`); i18n confines catalog and param lookups to own properties so `{constructor}` / `"constructor"` keys cannot reach the prototype chain (`packages/i18n/src/i18n.ts:31-35`, `interpolate.ts:17`).

## Simplicity

- **P1** — `packages/mailing-lists/src/mailing-lists.ts:87-94` — double-opt-in is half the dance: `subscribe` mints the token but the package never sends the confirmation email, and there is no hook to do so. Every app must hand-roll the confirm mailer, the URL scheme, and the resend path — the exact boilerplate a "Ghost-style" battery exists to delete. **Recommend:** accept a `confirmationMailer: { name, confirmUrl(token) }` option and enqueue it inside `subscribe`; same for an unsubscribe-footer convention on `broadcast`.
- **P1** — `packages/mail/src/mailer.ts:28-45` — `Email`/`RenderedEmail` are html-only: no `text` part, no `headers`, no `cc`/`bcc`/`replyTo`. Two concrete costs: html-only mail scores worse with spam filters, and `List-Unsubscribe`/`List-Unsubscribe-Post` headers are impossible to set — Gmail/Yahoo have required one-click unsubscribe for bulk senders since 2024, so `@volo/mailing-lists` broadcasts are undeliverable-at-scale by construction. **Recommend:** add `text?` and `headers?: Record<string,string>` to both shapes (header values CRLF-validated per the Security P1), and have mailing-lists set `List-Unsubscribe` automatically.
- **P1** — `packages/i18n/src/i18n.ts:73-77` — `plural` hardcodes English rules (`count === 1 ? "one" : "other"`). French (0 is singular), Russian/Polish (few/many), and Arabic (six categories) all render wrong; for a package whose pitch is pluralization this is a correctness bug, not a feature gap. **Recommend:** use `Intl.PluralRules` (zero-dep, present in Node and Workers) keyed by the locale argument, supporting `zero/one/two/few/many/other` suffixes with fallback to `other`.
- **P2** — `packages/feeds/src/rss.ts:47-49` and `atom.ts:47-56` — both builders advertise "valid RSS 2.0 / Atom 1.0" (docstrings at `rss.ts:34-40`, `atom.ts:34-40`) yet treat spec-required fields as optional: RSS 2.0 requires channel `<description>`; Atom requires feed and entry `<id>` and `<updated>`. Dates also pass through unvalidated (RSS wants RFC 822, Atom wants RFC 3339) — `published: "yesterday"` emits silently. **Recommend:** require the spec-mandated fields in the types (or synthesize defaults) and validate/format dates from `Date` inputs.
- Healthy: all five packages are admirably small (largest source file is 160 lines), dependency-light (feeds/seo/i18n have zero runtime deps), closure-factory/explicit-db shaped per the style direction, and every error is coded. This is the right architecture; the gaps are completeness, not design.

## Durability

- **P1** — `packages/mailing-lists/src/mailing-lists.ts:116-126` — `broadcast` is not crash-safe: it enqueues N jobs in a sequential loop with no surrounding transaction, no broadcast record, and no idempotency key (`@volo/queue`'s `enqueue` at `packages/queue/src/queue.ts:193-203` offers none). A crash after recipient k leaves no record of progress; re-running the broadcast double-sends to the first k recipients. **Recommend:** a `broadcasts` table plus per-recipient delivery rows (or a unique `(broadcastId, subscriberId)` job key once the queue grows one), so re-running resumes instead of repeating.
- **P2** — `packages/mail/src/mailer.ts:95-114` — delivery itself rides the queue's at-least-once semantics (claim bumps `attempts` up front, `queue.ts:240`; reclaim after a worker death, `queue.ts:213`), so a crash *after* `transport.send` but before ack redelivers — duplicate email with no dedupe handle. Inherent to the model, but undocumented. **Recommend:** document at-least-once on `MailTransport`, and pass a stable job-derived `messageId` in `RenderedEmail` so idempotent providers can dedupe.
- **P2** — `packages/mail/src/mailer.ts:75,96-102` — mailer builders are in-process registrations; a worker deployment missing one `define` burns all five attempts as `MAIL_UNKNOWN_MAILER` and the job dies. Deploy-skew hazard for the exact "deploy-safe reclaim" story the docstring sells (`mailer.ts:8-9`). **Recommend:** treat `MAIL_UNKNOWN_MAILER` as non-retryable-but-parked (or document that workers must register the superset of mailers).
- Verified solid: retries/backoff are real — default 5 attempts (`packages/queue/src/queue.ts:43,203`), exponential backoff (`queue.ts:420-421`), terminal failure with recorded `lastError` (`queue.ts:389-417`), all exercised by `packages/mail/test/mailer.test.ts:80-104`. The request path never blocks on delivery.

## Observability

- **P1** — `packages/mail/src/mailer.ts` (whole file) — the money path emits nothing: no log, no metric, no event when an email is rendered, sent, retried, or terminally failed. Operators can only forensically poll queue rows. A launch with invisible email delivery means "did the password reset go out?" is unanswerable. **Recommend:** an `onDelivered`/`onFailed` hook pair (or `@volo/observability` spans around `deliver` and `transport.send`) carrying mailer name, job id, and attempt — never the recipient body.
- **P2** — `packages/mailing-lists/src/mailing-lists.ts:116-126` — `broadcast` returns only a count; no broadcast id, no per-recipient enqueue trace, no progress signal for long fan-outs. Folding into the `broadcasts` table from the Durability P1 gives this for free. **Recommend:** return `{ broadcastId, enqueued }` and emit one span per broadcast.
- **P2** — feeds/seo/i18n are pure functions and rightly silent; the one gap is i18n missing-key visibility: `t()` returns the key itself (`packages/i18n/src/i18n.ts:62`) which is good UX-failure signaling, but nothing counts misses. **Recommend:** an optional `onMissing(locale, key)` callback so apps can meter translation gaps in production.

## Top launch-blockers

1. **P0 — `@volo/mail` cannot send email.** `MailTransport` (`packages/mail/src/mailer.ts:43-45`) has zero implementations in the repository — a full-repo search finds no SMTP, Resend, SES, Postmark, Sendgrid, Mailgun, or Workers-email transport; the only transports are in-memory test doubles (`packages/mail/test/mailer.test.ts:13-17`, `packages/mailing-lists/test/mailing-lists.test.ts:79-83`). Identity's verify/reset flows and mailing-lists' entire purpose dead-end at an interface. Ship at least one production transport (an SMTP transport for Node + a fetch-based provider transport that also runs on Workers) before any launch claim involving email.

(No other P0s. The P1 cluster — header-injection validation, confirmation-email send, broadcast crash-safety, `List-Unsubscribe` capability, plural rules, delivery observability — is the pre-1.0 gate.)

## Recommended plan items

1. **Ship a real transport** (P0): `@volo/mail-smtp` (Node) and a fetch-based provider transport (Resend/SES-API shaped) that runs on Workers; integration-test against a local SMTP sink. Everything else in the domain is moot until this lands.
2. **Harden the Mailer contract** (P1): CRLF rejection on `to`/`subject`/`from` (+ future headers) with coded errors; add `text?` and `headers?` to `Email`/`RenderedEmail`.
3. **Finish double-opt-in** (P1): `subscribe` enqueues the confirmation email via a configured mailer + URL builder; rotate the token on confirm; unique `(list_id, email)` upsert and email-shape validation.
4. **Make broadcast a first-class, resumable entity** (P1): `broadcasts` table + per-recipient rows, idempotent re-run, `List-Unsubscribe` header set automatically, chunked enqueue in a transaction, return `{ broadcastId, enqueued }`.
5. **Delivery observability** (P1): `onDelivered`/`onFailed` hooks or observability spans around `deliver`; broadcast span; i18n `onMissing` counter.
6. **Correct i18n pluralization** (P1): swap `count === 1` for `Intl.PluralRules` with category-suffixed keys and `other` fallback.
7. **Spec-tighten feeds & seo** (P2): require RSS channel description and Atom id/updated (or synthesize), format dates from `Date`, strip newlines in robots.txt directives, add token + `(list_id, status)` indexes to the mailing-lists migration.
