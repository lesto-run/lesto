# Core Runtime & HTTP — Architecture Review

## Domain summary

This domain is Lesto's request spine: `@lesto/errors` (coded errors + Result), `@lesto/router` (pattern compile/match, two routers), `@lesto/web` (the transport-free dispatch core — legacy Controller/Application stack plus the code-first `lesto()` builder, page streaming, hardening helpers, ALS request context), `@lesto/kernel` (assembly: migrations + dispatch + `secureStack`), and `@lesto/runtime` (the node:http front door: body bounds, timeouts, trust-proxy, ETag/304, static/site dispatch, SQLite seam). Contrary to this repo's history of aspirational docs, this domain is **real**: every package has working source, 100%-coverage-thresholded tests, and the node path is unusually well hardened for a young framework (per-request error boundary, socket limits, traversal defense, ReDoS refusal at route compile). The launch risks are not missing plumbing — they are *contract-level* flaws that get expensive after 1.0: a shared mutable 404/400 singleton that can leak headers across requests, a header map that cannot represent two `Set-Cookie`s, percent-encoded params handed raw to handlers, a spoofable `trustProxy: true` semantics, and two whole dispatch stacks shipped side by side. Verdict: **launchable after one P0 fix; not 1.0-freezable until the contract P1s land.**

## Performance

- **P1 — no response compression anywhere on the node path.** `applyResponse` (packages/runtime/src/response.ts:144–164) writes string/bytes/stream bodies verbatim; nothing in `serve()` (packages/runtime/src/server.ts:601–653) negotiates `Accept-Encoding`. Streamed HTML pages and `text/javascript` client bundles go out uncompressed unless an external proxy compresses — but the runtime's own docs position it as a public tier (server.ts:128–131). Recommendation: add opt-in (default-on) brotli/gzip for buffered bodies and a `CompressionStream`/zlib transform for the stream arm, skipping already-encoded and image types.
- **P2 — routing is a linear regex scan per request.** `RouteTable.match` (packages/router/src/table.ts:63–79) and legacy `Router.resolve` (packages/router/src/router.ts:142–161) iterate every entry, running `exec` on each method-matching pattern. Fine at tens of routes; a content-heavy app (the WordPress pitch) will register hundreds. Recommendation: bucket by method, exact-match static paths via a `Map`, keep the regex scan only for parameterized patterns.
- **P2 — every request body is fully buffered before dispatch, and >1 MiB is impossible.** `readBody` (packages/runtime/src/server.ts:324–368) buffers the whole body; `maxBodyBytes` defaults to 1 MiB globally (server.ts:244). There is no streaming-request path and no per-route override, so file upload endpoints cannot exist without raising a global cap that then applies to every anonymous POST. Recommendation: per-route body policy (limit + streaming opt-in) once route metadata exists; until then document the cap loudly.
- **P2 — the dynamic ETag/304 machinery is dead on the primary page path.** `withEtag` skips streams (packages/runtime/src/server.ts:485–491), and `lesto()` pages always stream (packages/web/src/render-page.tsx:213–219), so conditional GET only ever fires for buffered `c.html()` responses. Not a bug — but the Tier-0 "no-cache + 304 for HTML" claim (http-cache.ts:1–17) is mostly aspirational for real apps. Recommendation: either buffer-below-threshold pages to re-enable 304s, or re-scope the claim.
- **P2 — buffered responses ship chunked, with no `Content-Length`.** Nothing sets `Content-Length` for string/bytes bodies (response.ts:148–163), forcing chunked transfer encoding overhead and denying clients progress information. Recommendation: set `Content-Length` for the two buffered arms in `applyResponse`.

## Security

- **P0 — shared mutable 404/400 response singletons can leak headers across requests.** `NOT_FOUND` is a module-level constant returned by reference to every unmatched request (packages/web/src/lesto.ts:90–97, returned at lesto.ts:347 via `notFoundHandler`); `BAD_REQUEST` likewise for every params-validation failure (packages/web/src/render-page.tsx:84–88, 178). `LestoResponse` fields are mutable, and the middleware contract explicitly invites response post-processing ("delegate and then adjust the result (add headers…)", packages/web/src/middleware.ts:27–30). Any app middleware that does the ubiquitous Express idiom `const res = await next(); res.headers["set-cookie"] = …; return res;` permanently mutates the singleton — every later 404 for **every user** carries that header (a session cookie, a CSP override, a request id). In-tree middleware happens to spread (`cors`'s `{ ...response }`), but the framework cannot control app code. Recommendation: return a freshly constructed object per request (a one-line `notFound()` factory), and `Object.freeze` any constant responses in dev as a tripwire.
- **P1 — `trustProxy: true` (and the predicate form) believes the attacker-controlled left-most `X-Forwarded-For` entry.** `clientFromChain` returns `chain[0]` for `true`/predicate policies (packages/runtime/src/trust-proxy.ts:95–105, 129–131). A client behind your one trusted LB sends `X-Forwarded-For: 1.2.3.4`; the LB appends the real address, producing `1.2.3.4, <real>`; Lesto resolves the client as `1.2.3.4`. That defeats exactly what the context IP is for — rate-limit keying (`secureStack` rate-limit keys on `currentContext()?.ip`, packages/ratelimit/src/middleware.ts:103–105) and audit logs. The numeric hop-count path (`length - 1 - n`) is correct; the most ergonomic settings are the spoofable ones, and the in-repo CVE review checkmarked this module as solved (docs/readiness/SERVER-STREAMING-CVE-REVIEW-2026-06-09.md:233). Recommendation: make `true` mean "one trusted hop" (resolve `chain[length - 1]`), make the predicate peel right-to-left while the predicate accepts, and reserve left-most for an explicit `"all"` escape hatch.
- **P1 — `Record<string, string>` headers cannot represent multiple `Set-Cookie`s.** The dispatch contract (packages/web/src/types.ts:80–87) and every writer (`applyResponse` response.ts:148, the 304 path, the edge adapter seam) carry headers as a single-valued map. Setting a session cookie *and* a flash/CSRF cookie in one response silently clobbers one of them; merging in middleware (`{ ...headers, ...response.headers }`, cors middleware.ts:51) makes the loss invisible. This is an API-shape flaw that becomes a breaking change after 1.0. Recommendation: widen the body contract now — `Record<string, string | string[]>` or a dedicated `setCookies: string[]` field — and write the array arm in `applyResponse`/`respondNotModified`.
- **P1 — route params are delivered percent-encoded, undecoded.** Captures are copied raw from the path (packages/router/src/table.ts:71–78, router.ts:150–157); no `decodeURIComponent` exists anywhere in router/web/runtime (verified by grep). `c.param("slug")` for `/posts/hello%20world` is `"hello%20world"`; a param containing `%2F` smuggles a path separator past the `[^/]+` segment matcher. Every app will decode ad hoc, inconsistently, and some will decode after using the value in a comparison or path join — the classic encoded-traversal hazard the framework should absorb. Recommendation: `decodeURIComponent` each capture at match time (rejecting malformed sequences as a 400 via a coded error), matching Rails/Express semantics; `pathFor` already encodes on the way out (router.ts:187).
- **P2 — the "headers are lowercased" contract is implicit at the dispatch seam.** `Lesto.handle`/`Application.handle` accept `options.headers` verbatim (packages/web/src/lesto.ts:333–343, application.ts:69–81); `c.header()` lowercases only the lookup name (handler-context.ts:83–85). The node transport normalizes (request.ts:48–62) and Fetch `Headers` iterate lowercased, but a test, MCP, or future adapter passing `Origin:` makes the CSRF origin-check silently see "no header". Recommendation: normalize keys once inside `handle()` — it is the seam every transport funnels through.
- **P2 — no CSP by default, and no nonce machinery to ever enable one.** Deliberate and documented (packages/web/src/harden.ts:53–61; server.ts:162–172) because island bootstrap inlines `<script>` JSON — but that means the recommended posture (`RECOMMENDED_CSP`) is unusable by any app with islands. Recommendation (pairs with the islands domain): per-request nonce minted in the runtime, threaded through the render context, stamped onto inline island scripts, then flip `RECOMMENDED_CSP` to a default-on report-only.

## Simplicity

- **P1 — two complete dispatch stacks ship side by side.** Legacy `Router` + `Application` + `Controller` (packages/web/src/application.ts, controller.ts; packages/router/src/router.ts) and the target `lesto()` + `Context` + `RouteTable` coexist; the kernel is dual-mode (`"app" in config`, packages/kernel/src/kernel.ts:125–144) and both surfaces are exported from `@lesto/web`'s index (index.ts:18–27). ADR 0004 Phase 7.6 (deletion) is explicitly deferred. Shipping 1.0 with two router vocabularies, two contexts, and a string-target `"posts#index"` convention doubles the docs, the test matrix, and every future middleware decision. Recommendation: complete Phase 7.6 before 1.0 — delete `Application`/`Controller`/legacy `Router`, collapse `createApp` to `LestoAppConfig` only.
- **P1 — `@lesto/hooks` and `@lesto/config` are orphans the architecture doc sells as pillars.** Zero packages import either (verified across all package.jsons and sources); ARCHITECTURE.md line 3 claims "an extensibility model (hooks/plugins/themes) … built in". Honest options only: wire `Hooks` into the kernel boot/request lifecycle (named lifecycle hooks, plugin registration) or excise both packages and the claim until the plugin story is real. Carrying dead public surface into 1.0 is the worst of both.
- **P2 — `loadConfig` claims "typed configuration" but erases the schema.** It returns `Record<string, ConfigValue>` (packages/config/src/config.ts:68–71): `config.port` is `string | number | boolean` and field-name typos type-check. Recommendation (if config survives the orphan decision): a mapped conditional type `{ [K in keyof S]: FieldValue<S[K]> }` — table stakes for a TS-first framework.
- **P2 — `statusForError` hard-codes cross-package error codes as string literals.** `"RUNTIME_INVALID_JSON"` etc. live in `@lesto/web` (harden.ts:127–136) while the codes are declared in `@lesto/runtime` (errors.ts:13–22) — a rename compiles fine and silently turns a 400 into a 500. Recommendation: carry an `httpStatus` hint on `LestoError` itself (or a shared codes module in `@lesto/errors`).
- **P2 — `singularize` strips one trailing "s".** `resources("statuses")` yields member route name `statuse` (packages/router/src/router.ts:56). Cosmetic but it leaks into `pathFor` names. Dies with the legacy router if Phase 7.6 lands; otherwise document.

## Durability

- **P1 — boot-time migrations race under the documented multi-instance deployment.** `createApp` unconditionally runs pending migrations on boot (packages/kernel/src/kernel.ts:112–121) while `serve()`'s own docs promise "multiple instances scale horizontally and deploys are rolling restarts" (packages/runtime/src/server.ts:592–595). Two instances booting concurrently against one Postgres both run the same DDL — no advisory lock, no single-runner story. Fine for single-node SQLite; a deploy-time incident generator for the Postgres tier. Recommendation: take a `pg_advisory_lock` (or migrator-owned lease table) inside `Migrator.migrate`, and/or support `migrations: "skip"` so fleets migrate from a release job.
- **P1 — the handler timeout abandons work with no cancellation signal.** On overrun, `withTimeout` answers 503 and leaves the handler running forever (packages/runtime/src/server.ts:379–402, 843–851); `context.signal` aborts only on client disconnect (server.ts:762–777), and after the 503 is written `writableFinished` is true so it never fires. A wedged downstream (slow DB, hung fetch) accumulates zombie handlers holding connections and memory — the exact overload scenario the timeout exists for. Recommendation: race an internal `AbortController` with the deadline and abort `context.signal` on timeout so well-behaved handlers (and the page renderer, which already chains the signal — render-page.tsx:213–218) actually stop.
- **P2 — the streamed-page render deadline is a hardcoded 10s.** `RENDER_DEADLINE_MS` (packages/web/src/render-page.tsx:100) is not configurable per app or per route, and is independent of the transport's 30s `handlerTimeoutMs` — two unconnected budgets for one request. Recommendation: thread it through `lesto()`/`ServeOptions` with the constant as default.
- **P2 — `serve()` permanently mutates process-global state.** `installProcessSafetyNet` registers an irrevocable `unhandledRejection` listener (packages/runtime/src/server.ts:962–992) as a side effect of booting any server, switching the whole process from fail-fast to log-and-continue — including for rejections from non-Lesto code. Defensible default, but it should be a visible, revocable option (`safetyNet: false`) rather than ambient.
- **P2 — `Hooks.doAction` lets one throwing listener abort the chain.** Listeners run sequentially and an exception propagates (packages/hooks/src/hooks.ts:92–101) — for a WordPress-style extension surface, one bad plugin silencing all later listeners is the failure mode that ecosystem spent a decade containing. Moot while orphaned; decide isolation semantics (per-listener catch + report) before wiring it into the kernel.

## Observability

- **P1 — a mid-stream body failure is completely invisible.** When a streamed page's source errors, `pipeStream` destroys the socket and resolves silently (packages/runtime/src/response.ts:113–118); the access line was already emitted, and for streams it is logged at write-start with the pre-stream status (server.ts:870, 884) — so a render that died at byte 10,000 of 100,000 shows up as a clean 200 with nothing in `logError`. Truncated-HTML bugs become unreproducible ghost reports. Recommendation: give `applyResponse`/`pipeStream` an error sink (the runtime already injects `logError`) and record stream completion/truncation, ideally as a `lesto.stream_truncated` span attribute too.
- **P2 — early-rejected requests are access-logged with the wrong request line.** `handle` reads the body *before* computing method/path (packages/runtime/src/server.ts:824–827), so a 413 (or socket error) logs the defaults `GET /` (initialized at server.ts:819–820) — precisely the abusive traffic you most want attributed. Recommendation: call `requestLineOf` before `readBody`.
- **P2 — the request id never reaches the client, and inbound ids are ignored.** `newRequestId` always mints (server.ts:803–805); no `X-Request-Id` response header is set and no inbound `x-request-id`/`traceparent` is adopted, so a user bug report cannot be joined to a log line, and Lesto breaks any upstream trace context. Recommendation: echo `X-Request-Id` on every response; adopt (validated) inbound ids behind the same trust gate as `trustProxy`.
- **P2 — the default access log is an unstructured `console.log` line.** server.ts:953–955. The seam (`logRequest`) is right, but the default should be structured (JSON) so day-one deployments are queryable; `@lesto/observability` exists per ADR work yet the runtime defaults don't use it.
- **P2 — the readiness probe is unbounded.** `healthResponse` awaits `isReady()` with no timeout (server.ts:425–431); a hung DB ping makes `/readyz` hang instead of failing, which some orchestrators treat as ready-stale rather than not-ready. Recommendation: race the probe against ~1s and answer 503 on overrun.

## Top launch-blockers

1. **P0 — Shared mutable `NOT_FOUND`/`BAD_REQUEST` singletons** (packages/web/src/lesto.ts:90–97; packages/web/src/render-page.tsx:84–88). A one-line app middleware mutation leaks headers — including `Set-Cookie` — across requests and users, framework-wide. The fix is trivial (per-request factories + freeze constants); the consequence class (cross-user data leak) is the worst one. Fix before any launch traffic.

That is the only P0 in this domain. The nine P1s above are pre-1.0, not pre-launch — with the caveat that the **Set-Cookie multimap** and **param decoding** P1s are breaking API changes and get strictly more expensive every week they wait.

## Recommended plan items

1. **Kill the singletons** — `notFound()`/`badRequest()` factories, `Object.freeze` on any remaining constant responses, plus a regression test that mutates a 404's headers through middleware and asserts the next request is clean. (Hours; do first.)
2. **Widen the header contract for repeated headers** — `LestoResponse.headers: Record<string, string | string[]>` (or `setCookies`), update `applyResponse`, `respondNotModified`, `withSecurityHeaders` merges, and the edge-adapter seam in the same change. Breaking; land before any external consumer exists.
3. **Decode route params at match time** — `decodeURIComponent` in `RouteTable.match`/`Router.resolve` with a coded 400 on malformed input; tests for `%2F`, `%2e%2e`, unicode slugs.
4. **Fix `trustProxy: true`/predicate to right-most-trusted resolution**, with a test asserting the prepended-XFF spoof fails; document the chain semantics in `ServeOptions`.
5. **Timeout cancellation** — abort the request context's signal when `handlerTimeoutMs` fires; unify the page-render deadline with it and make both configurable.
6. **Observability batch** — stream-truncation reporting in `pipeStream`, request line before body read, `X-Request-Id` response header, structured default access log, bounded readiness probe. (One coherent PR over server.ts/response.ts.)
7. **Compression** — Accept-Encoding-aware brotli/gzip in `applyResponse` (buffered arms) + zlib transform for streams, content-type allowlist, off for already-encoded types.
8. **ADR 0004 Phase 7.6 — delete the legacy stack** — remove `Application`/`Controller`/legacy `Router`, collapse `createApp` to the `lesto()` shape, migrate the kernel/CLI/MCP tests still constructing `new Router()`.
9. **Decide the orphans** — wire `Hooks` into kernel lifecycle (with per-listener error isolation) and give `loadConfig` mapped-type inference, **or** delete both packages and amend ARCHITECTURE.md. No third option.
10. **Migration lease** — advisory lock (PG) / single-runner guard in the migrator, `migrations: "skip"` boot mode for fleets; document the rolling-deploy story `serve()` already promises.
