# Data & Persistence — Architecture Review

## Domain summary

This domain is Keel's "one substrate" promise: a typed query layer (`@keel/db`) over an async driver seam (ADR 0006), a Postgres adapter (`@keel/pg`), migrations (`@keel/migrate`), and the DB-backed batteries (queue, cache, workflows, pubsub, storage, admin) that justify "production = one Postgres." The query layer is genuinely good — parameterized end-to-end, immutable chains, cross-driver parity proven in CI against a real Postgres service — and the queue's at-least-once design on SQLite is real and tested. But the headline promise is **not** deliverable today: every DDL emitter except the brand-new ADR 0013 stores is SQLite-only (`AUTOINCREMENT`, int4-overflowing epoch-ms), so migrations and the queue/cache schemas cannot install on Postgres; and the queue's claim statement double-delivers under concurrent Postgres workers because the promised `FOR UPDATE SKIP LOCKED` was never written. Workflows ship step memoization, not the "crash-safe resume" the package description claims. Verdict: **launchable as SQLite-single-node only; the Postgres production story is blocked by two P0s**, and the domain has zero tracing/metrics despite an in-repo OTLP tracer.

## Performance

- **P1** — `packages/admin/src/admin.ts:180` — `list()` runs `db.select().from(table).all()` with no limit, offset, or column projection; an admin page over a 1M-row table hydrates the entire table into memory. Recommendation: add `limit`/`offset` (the `SelectQuery` already supports both) with a default page size, and project only `fields` + pk instead of `SELECT *`.
- **P1** — `packages/queue/src/queue.ts:33-55` — `done`/`failed` rows accumulate forever (no pruning/retention), and the claim index `(status, queue, run_at)` indexes every dead row. Recommendation: a `prune(olderThanMs)` API + scheduler wiring, and (on PG) a partial index `WHERE status = 'ready'`.
- **P2** — `packages/queue/src/queue.ts:255-263` — every `runOnce()` issues a `reclaim()` UPDATE before every claim, so an idle worker performs two writes per poll (default 200ms) per concurrency slot. Recommendation: run reclaim on its own cadence (e.g. every Nth poll or `visibilityMs/2`).
- **P2** — `packages/cache/src/sql-store.ts:39-49` + `packages/cache/src/cache.ts:165-179` — expired entries are deleted only when read; keys never re-read accumulate unboundedly. Recommendation: a `sweep(now)` on the store (the ratelimit store already has one — `packages/ratelimit/src/sql-store.ts`) wired to the scheduler.
- **P2** — `packages/storage/src/file.ts:56-62` — `list(prefix)` recursively walks the entire root then filters in JS; O(total objects) per call. Recommendation: descend only into the prefix's directory subtree.

## Security

- **P1** — `packages/db/src/queries.ts:394-395, 418-420` — `Db.exec(sql: string)` is the only raw escape hatch and takes **no parameters**, while the condition vocabulary lacks `gt/lt/gte/lte/in/like` (`packages/db/src/conditions.ts` exports only `eq/ne/isNull/isNotNull/and/or`). Every query the DSL can't express funnels users toward string-interpolated SQL — the exact injection class the layer's design brags about preventing (`queries.ts:17-19`). Recommendation: add a parameterized `db.raw(sql, params)` (or expose `prepare`) and grow the condition set; lint-warn on `exec` with interpolation.
- **P1** — `packages/admin/src/admin.ts:129-237` — admin mutations (`create/update/destroy`) have no audit logging and no authz seam; the layer assumes the HTTP caller gates everything, and a compromised or misconfigured route silently gets unattributed full CRUD. Recommendation: accept an optional `onMutation(event)` hook (actor, resource, id, patch) and document that authn/authz is mandatory upstream; pre-launch, wire estate's admin routes through it.
- **P2** — `packages/migrate/src/schema.ts:24, 48, 66` and `packages/migrate/src/table-builder.ts:54` — table/column/index identifiers are interpolated unquoted, unlike `@keel/db`'s `quoteIdentifier` discipline (`packages/db/src/identifier.ts:18-26`). Inputs are migration-author-controlled so risk is low, but a column named with a space or keyword breaks, and the inconsistency invites copy-paste into riskier contexts. Recommendation: route all DDL identifiers through one quoting function.
- **P2** — `packages/cache/src/sql-store.ts:63` — `set()` of an `undefined` value produces `JSON.stringify(undefined) === undefined`, which the driver binds as a raw error (better-sqlite3 throws a non-Keel TypeError). Recommendation: coalesce to JSON `null` like workflows does (`packages/workflows/src/engine.ts:95-100`) or throw a coded error.
- Positive: every user value in db/queue/cache/workflows/admin rides a placeholder; `FileBackend.resolve` (`packages/storage/src/file.ts:93-100`) rejects `..` and absolute keys; identifier NUL-byte guard exists (`db/src/identifier.ts:19`).

## Simplicity

- **P1** — `packages/orm/` (entire package, ~900 lines src + 433 test) — legacy per ADR 0004, sync seam incompatible with ADR 0006, **zero in-tree consumers** (verified: only `create-keel/test/scaffold.test.ts:113-114` references it, to assert its absence). It still runs in the serial coverage gate on every CI pass. Recommendation: delete it (ADR 0004 Phase 7.6); "preserved for out-of-tree apps" is moot pre-1.0 with no published versions.
- **P1** — two parallel DDL systems: `@keel/migrate`'s string-building `TableBuilder` (`table-builder.ts:43-105`) vs `@keel/db`'s schema-as-value `createTableSql` (`ddl.ts:57-61`). ADR 0004 declares the schema value the single source of truth, yet the Rails-style builder survives and both now need the same dialect work twice. Recommendation: when building the dialect layer (P0 below), do it once in `@keel/db` and reduce `migrate` to ordering/bookkeeping + `s.execute(createTableSql(table, dialect))`.
- **P2** — the `SqlDatabase`/`SqlStatement` seam is re-declared in at least 8 places (`db/src/sql.ts`, `migrate/src/types.ts:14-65`, `cache/src/types.ts:54-71`, `queue/src/types.ts:86-111`, `workflows/src/types.ts:16-30`, plus kernel/auth/ratelimit) — ADR 0006 itself counts them. Structural typing keeps it working, but a seam change is now an 8-file ripple with 8 doc comments to keep truthful. Recommendation: one canonical type in `@keel/db` (or a zero-dep `@keel/sql-seam`), re-exported.
- **P2** — `packages/migrate/src/table-builder.ts:91` — `references("category")` emits `REFERENCES categorys(id)`; the naive `${name}s` pluralization is a footgun the ORM's real inflector (orm/src/inflector.ts) was built to avoid. Recommendation: take an explicit `{ table }` option; drop the guess.

## Durability

- **P0** — dialect drift: the DDL substrate is SQLite-only, so **Postgres is not real end-to-end**. Evidence: `packages/db/src/ddl.ts:36` (`AUTOINCREMENT`), `packages/migrate/src/table-builder.ts:51` (`id INTEGER PRIMARY KEY AUTOINCREMENT` on every table), `packages/queue/src/queue.ts:36` (jobs table AUTOINCREMENT), `packages/cache/src/sql-store.ts:17` (`expires_at INTEGER` — epoch-ms ~1.8e12 overflows PG int4; the ADR 0013 stores explicitly use `BIGINT` for exactly this reason, `packages/ratelimit/src/sql-store.ts:28-30`). The CI parity suite dodges this by hand-writing per-dialect DDL (`packages/integration/test/db-parity.integration.test.ts:35-40`; acknowledged again at `durable-stores.integration.test.ts:12-16`). Consequence: `migrator.migrate()`, `queue.installSchema()`, and `installCacheSchema()` all fail on a real Postgres, which falsifies ARCHITECTURE.md's "production = one Postgres… the in-house APIs never change" (lines 25-28). Recommendation: a `dialect` parameter on `createTableSql`/installers (`AUTOINCREMENT`→`GENERATED ALWAYS AS IDENTITY`, `INTEGER` epoch columns→`BIGINT`), threaded the way ratelimit already threads `Dialect`; then run the schema installers in the PG CI leg.
- **P0** — `packages/queue/src/queue.ts:237-249` — the claim `UPDATE … WHERE id = (SELECT … LIMIT 1) RETURNING *` is atomic on SQLite but **double-claims on Postgres**: under READ COMMITTED, two workers' subselects pick the same id; the blocked worker's EvalPlanQual recheck still passes (`id` is unchanged), so both update and both receive the job. The file's own header (queue.ts:28-29) and ARCHITECTURE.md:100 promise `FOR UPDATE SKIP LOCKED`; it was never implemented. Recommendation: thread `dialect` into `Queue` and emit `SELECT … FOR UPDATE SKIP LOCKED` inside the subselect on PG; add a concurrent-claim test to the PG CI leg (mirroring the rate-limit atomicity proof at `durable-stores.integration.test.ts:173-199`).
- **P1** — `packages/queue/src/queue.ts:376-418` — `complete()` and `fail()` update `WHERE id = ?` with no fencing on `status`/`locked_until`. A worker that stalls past its visibility deadline (GC pause, slow handler) and then *fails* will set a job another worker already completed back to `'ready'` — a **finished job re-runs**, and a failed terminal state can overwrite a `done` one. Recommendation: guard terminal transitions with `AND status = 'running' AND locked_until = ?` (the claim's lock value as fencing token) and treat 0 changes as "lost the claim, do nothing."
- **P1** — `packages/migrate/src/migrator.ts:62-94` — no cross-process migration lock: two instances booting simultaneously (rolling deploy) both compute `pending` from the same snapshot and race the DDL; one crashes on `CREATE TABLE` collision mid-deploy. Recommendation: `pg_advisory_lock` on PG / rely on the FIFO single connection on SQLite, behind a seam method.
- **P1** — workflows' durability claim is overstated. `packages/workflows/src/engine.ts:35-122` implements step memoization only: there is no run journal (no status/started/completed rows), nothing re-invokes an interrupted run (`run()` must be re-called with the same runId by the caller), `sleep` is a plain timer (`packages/workflows/src/sleep.ts:9-12` — a crash during a 3-day sleep resumes only if something re-runs the workflow, and then re-sleeps from zero), and `waitForEvent` (ARCHITECTURE.md:76) doesn't exist. Recommendation: add a `keel_workflow_runs` journal (status, input, name) + a queue-backed resume driver (enqueue `workflow:run` jobs so the queue's at-least-once does the re-invocation), and memoize sleep deadlines as steps; until then, fix the package description ("crash-safe resume" → "resumable step memoization").
- **P1** — `packages/queue/src/scheduler.ts:60-72, 120-141` — cron/interval dedupe state (`lastMinuteKey`, `lastRunAt`) is in-process memory: two app instances each enqueue every cron (duplicate jobs fleet-wide), and a restart loses interval phase / misses crons during downtime with no catch-up. Recommendation: persist last-fired per entry in a table with an atomic claim (one `UPDATE … WHERE last_minute < ?` gate), or document single-scheduler-instance as a hard deployment constraint.
- **P1** — `packages/storage/` has no production backend: memory dies with the process, `FileBackend` requires a persistent local disk and `node:fs` — neither exists on Cloudflare Workers (ADR 0002's target) or in a multi-node fleet. Recommendation: an S3/R2 backend over `fetch` + SigV4 (no SDK) is the smallest credible v1.
- **P2** — `packages/queue/src/queue.ts:74-91` + `claim()` — a poison row whose `payload` fails `JSON.parse` throws inside `hydrate` *after* the claim UPDATE, before `fail()` can ever run; the job loops claim→throw→reclaim forever, bypassing `maxAttempts` (which is only enforced in `fail()`, queue.ts:393). Recommendation: parse defensively in `runOnce` and route parse failures through `fail()`.
- **P2** — `packages/workflows/src/engine.ts:78-82` — two concurrent executions of the same runId race the step INSERT; the loser gets a raw PK-violation from the driver, not a coded error, and the two executions may have produced different results. Recommendation: `INSERT … ON CONFLICT DO NOTHING` + re-read, making the journal the arbiter.
- **P2** — `packages/db/src/queries.ts:139-141` — offset-without-limit renders `LIMIT -1`, which Postgres rejects (`LIMIT` must be non-negative; the idiom is `LIMIT ALL`/bare `OFFSET`). The comment defers to "the driver's concern" but `@keel/pg`'s translator (`translate.ts`) only rewrites placeholders. Untested because the parity suite always pairs limit+offset (`db-parity.integration.test.ts:100`).
- **P2** — `packages/pubsub/src/pubsub.ts:79-86` — `publish` awaits listeners serially and a throwing listener rejects the publish, silently skipping all later subscribers. Recommendation: settle all listeners and aggregate failures, or document the fail-fast semantic loudly.

## Observability

- **P1** — the domain emits **no spans, no metrics, no log hooks** — while `packages/observability` ships a working OTLP tracer and ARCHITECTURE.md:116 promises "one trace spans UI → API/controller → DB query" and "queue/workflow spans." No data package depends on `@keel/observability` (verified in all ten package.jsons); `@keel/db` has no query-timing hook, the migrator applies schema changes with no event emission (`migrator.ts:84-94`), cache has no hit/miss counters, workflows no step timings. Recommendation: a lightweight optional `onQuery({ sql, durationMs })` seam on `createDb` and `onJob`/`onStep` hooks on queue/workflows — seams first, OTLP wiring second.
- **P1** — `packages/runtime/src/worker.ts:14-16` — `runWorker` forwards `concurrency` but **drops the `onError` seam**, and `Queue.work`'s reporter early-returns when absent (`queue.ts:304-307`), so the blessed production wire silently swallows every poll-loop fault (transient DB outages, claim errors). The queue built the observability seam; the runtime wire discards it. Recommendation: accept and forward `onError` (default: structured stderr log).
- **P2** — `packages/queue/src/queue.ts:353-363` — `stats()` casts `COUNT(*) AS n` to `number`, but node-postgres returns int8 as a **string**; on PG the dashboard numbers are `"3"` not `3`. `@keel/db.count()` already coerces (`queries.ts:197` `Number(row.c)`); do the same here. Also: no oldest-ready-job-age or depth gauge — the two numbers an operator actually pages on.
- **P2** — error codes are uniformly good (every refusal in db/migrate/queue/workflows/admin/storage is a coded `KeelError` subclass — genuinely better than most frameworks at this), but there is no central code registry; collisions across packages are unchecked. Recommendation: a build-time uniqueness check over the `*ErrorCode` unions.

## Top launch-blockers

1. **P0 — SQLite-only DDL blocks the entire Postgres story.** `createTableSql` (`db/src/ddl.ts:36`), `TableBuilder` (`migrate/src/table-builder.ts:51`), `queue.installSchema` (`queue.ts:36`), and `installCacheSchema` (`cache/src/sql-store.ts:17`, int4 overflow) all fail on Postgres. The framework's core pitch — SQLite dev → Postgres prod, same APIs — is currently false above the query layer. Fix: a `Dialect` parameter through the DDL renderers and schema installers (the pattern `@keel/ratelimit` already proves), then run every installer in the existing `db-parity-postgres` CI job.
2. **P0 — queue double-delivery on Postgres** (`queue.ts:237-249`): no `FOR UPDATE SKIP LOCKED`, so two PG workers can claim the same job. At-least-once degrades to routinely-twice under the exact multi-worker deployment Postgres exists to enable. Fix: dialect-aware claim SQL + a concurrent-claim proof in CI.

Both P0s are halves of one truth: **the dialect layer named as a follow-up in ADR 0006 is the launch gate for this domain.** Until it lands, launch honestly as SQLite-single-node.

## Recommended plan items

1. **Build the dialect layer** (`Dialect = "sqlite" | "postgres"` in `@keel/db`): dialect-aware `createTableSql` (identity columns, BIGINT epoch-ms), thread it through `migrate`, `queue.installSchema`, `installCacheSchema`, `installWorkflowSchema`; fix `LIMIT -1` rendering. Acceptance: every schema installer + migrator runs in the `db-parity-postgres` CI job. (Clears P0 #1.)
2. **Postgres-safe queue claim**: `FOR UPDATE SKIP LOCKED` on PG + fenced terminal transitions (`AND status='running' AND locked_until = ?` in `complete`/`fail`) + poison-payload routing through `fail()`. Acceptance: a 12-concurrent-workers claim test on PG admitting each job exactly once (mirror the rate-limit atomicity proof). (Clears P0 #2 + the resurrection P1.)
3. **Migration lock**: advisory lock on PG, documented single-connection FIFO on SQLite; concurrent-boot test.
4. **Observability seams**: `onQuery` on `createDb`, `onJob`/`onStep` hooks, forward `onError` through `runWorker`, coerce `stats()` on PG, add queue depth/oldest-age. Wire to `@keel/observability` in the kernel.
5. **Admin hardening**: pagination + field projection in `list()`, `onMutation` audit hook, map `DB_EMPTY_UPDATE` to an admin-coded error.
6. **Delete `@keel/orm`** and fold `TableBuilder` into schema-as-value DDL (migrate keeps ordering/bookkeeping only).
7. **Workflows v1 honesty + run journal**: `keel_workflow_runs` table, queue-backed resume driver, durable sleep-as-step; rename the claim until then.
8. **Scheduler multi-instance safety**: persisted, atomically-claimed cron firings (or a documented single-scheduler constraint enforced by the runtime).
9. **Storage S3/R2 backend** (fetch + SigV4) so the battery exists on Cloudflare and fleets; until then mark storage "local/dev only" in docs.
10. **Retention & sweeps**: queue `prune()`, cache `sweep()` (pattern exists in ratelimit), both wired to the scheduler; partial index on `keel_jobs(status='ready')` for PG.
11. **Pubsub direction call**: either build LISTEN/NOTIFY per ARCHITECTURE.md:80 or rescope the package as in-process events and update the architecture doc (which should also drop the nonexistent `test/durability-demo.js` citation).
