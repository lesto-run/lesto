# Edge, Deploy & Sites ‚Äî Architecture Review

## Domain summary

The edge tier is in materially better shape than its reputation: the prior P0 ("CF adapter bypasses all hardening") is **fixed and locked by tests** ‚Äî `toFetchHandler` runs the same shared hardening layer (`@volo/web/harden.ts`) the node server runs: security headers merged under every response, an error boundary mapping throws to coded statuses with internals-free bodies, a per-request `AsyncLocalStorage` context with `cf-connecting-ip`, and a streaming-bounded 1 MiB body cap (`packages/cloudflare/src/fetch-handler.ts:25-32, 309-431`; parity tests `packages/cloudflare/test/cloudflare.test.ts:284-417`). `@volo/sites` genuinely prerenders through the app's own `handle` (no second renderer). The honest gaps are around the edge, not in it: `@volo/deploy` never uploads to any cloud (the real Cloudflare deploy is `wrangler deploy`, bypassing `shipStatic`/`shipRelease` entirely), the wrangler generator cannot reproduce the committed `wrangler.jsonc` it claims to have generated, the example worker **falls back to a committed signing secret when `SESSION_SECRET` is unset** (the one P0), and the edge app mounts none of the security middleware its node twin runs. Verdict: ship-quality adapter, prototype-quality deploy story, one launch-blocking secret-handling bug.

## Package inventory

| package | maturity | one-line reality (verified) |
|---|---|---|
| `@volo/cloudflare` | built | Hardened fetch adapter + assets-first composition + wrangler generator; 100%-cov tests incl. hardening parity, streams, bytes; no ETag/304, no `waitUntil` plumbing; generator emits a config thinner than the one actually deployed. |
| `@volo/deploy` | partial | Pure planner (`planDeploy`) + local-disk uploader/release-store with atomic symlink flips and rollback ‚Äî solid logic, but **no cloud uploader exists** and the string-typed ship seam corrupts binary files; unused by the headline Cloudflare path. |
| `@volo/sites` | built | `defineSites` validation, prerender via the app's own `handle("GET", path)`, all-or-nothing build gate, path-escape-checked sink; string-bodied HTML pages only, no asset hashing/cache pipeline. |

## Performance

- **P2 ‚Äî `packages/cloudflare/src/assets.ts:43` ‚Äî redundant assets RPC on every dynamic request.** With an `assets` binding, Cloudflare serves matching static files *before* invoking the Worker, so any GET/HEAD that reaches the Worker has already missed the binding; `withAssets` then probes `assets.fetch` again and waits for a 404 before dispatching. One wasted round-trip per dynamic page view at the PoP. Recommendation: consult the deploy plan's routing rules (`packages/deploy/src/plan.ts:94-99`, currently unused on this path ‚Äî see Simplicity) to skip the probe under known-dynamic prefixes, or document the probe as a non-Cloudflare-portability shim.
- **P2 ‚Äî `packages/cloudflare/src/fetch-handler.ts:19-22` ‚Äî no ETag/304 for dynamic edge responses.** Acknowledged in-code: the node server's conditional GET (`packages/runtime/src/http-cache.ts`, wired at `server.ts:188-194`) is absent on the edge because the hash is `node:crypto`-sync. Every repeat view of an unchanged SSR page re-sends full HTML. Recommendation: async Web-Crypto SHA-256 ETag for string/bytes bodies in `toFetchHandler` (streams stay exempt, matching node).
- **P2 ‚Äî `examples/estate/src/production.ts:100` + `examples/estate/out/marketing/client.js` ‚Äî island bundle is unhashed (`client.js`), and the deploy plan has no cache-policy field (`packages/deploy/src/plan.ts:25-34` carries `contentType` only).** No `Cache-Control: immutable` story is possible; Cloudflare falls back to etag revalidation per request. Recommendation: content-hash bundle filenames at build (`@volo/assets` is already planned in ADR 0011) and add an optional cache policy to `PublishFile`.
- **Verified good:** per-isolate handler memoization keyed by secret keeps app construction out of the request path (`examples/estate/worker.ts:55-68`); the bounded body read cancels mid-stream at the cap rather than buffering then checking (`fetch-handler.ts:188-220`, proven by the pull-counting test `cloudflare.test.ts:142-166`); the worker bundle is Preact-aliased (`examples/estate/wrangler.jsonc:21-28`), keeping isolate cold-start small.

## Security

- **P0 ‚Äî `examples/estate/src/edge.ts:238-240` ‚Äî silent fallback to a committed signing secret.** `edgeSecret(env)` returns `"estate-demo-edge-secret"` when `env.SESSION_SECRET` and `process.env.SESSION_SECRET` are both unset. The comment says "loud demo fallback" but nothing logs, warns, or refuses ‚Äî a production `wrangler deploy` that skips `wrangler secret put SESSION_SECRET` ships a Worker whose session trust root is public in this repo, and ADR 0002 names that secret "the whole trust root for stateless sessions." Combined with the passwordless `?as=` sign-in (below), any visitor mints a valid session for any user ‚Äî and even with real auth, anyone can forge cookies offline. Recommendation: fail closed ‚Äî throw at first request when `env.SESSION_SECRET` is absent (allow the literal fallback only under an explicit `VOLO_DEMO=1`-style opt-in), and make this the framework's documented pattern for every secret-bearing Worker.
- **P1 ‚Äî `examples/estate/src/edge.ts:115-234` vs `examples/estate/src/app.ts:39` ‚Äî the edge twin mounts none of the security middleware the node twin runs.** `buildEdgeApp` has zero `.use(...)`: no `secureStack`, no origin-check CSRF, no rate limit; the node app mounts `secureStack({ originCheck: {} })`. `SameSite=Lax` on the cookie covers classic cookie-borne CSRF, but the sign-in POST is open to login-CSRF and the credential endpoint is unthrottled at the edge. This is exactly the twin-divergence failure mode the hardening layer was built to prevent ‚Äî it just moved up a layer from transport to app middleware. Recommendation: mount `fromRequestMiddleware(secureStack({ originCheck: {} }))` (and per-isolate rate limiting) in `buildEdgeApp`, and add an e2e asserting a cross-origin POST is refused on the edge.
- **P1 ‚Äî `examples/estate/src/edge.ts:201-216` ‚Äî passwordless `?as=<id>` impersonation sign-in is deployed to the edge.** Known pending from the launch-security review; on the flagship *deployable* example it's now reachable on the public internet, not just localhost. Recommendation: replace with `@volo/identity` (already shipped) before launch, or hard-disable the route unless a demo flag is set.
- **Resolved (was P0) ‚Äî `packages/cloudflare/src/fetch-handler.ts:309-431`.** The "CF adapter bypasses all hardening" finding from `docs/readiness/SERVER-STREAMING-CVE-REVIEW-2026-06-09.md:40` is closed: shared `securityDefaults`/`withSecurityHeaders`/`statusForError`/`bodyForStatus` from `@volo/web` run on every edge response including 500s (test `cloudflare.test.ts:351-364` proves a thrown secret never reaches the wire and the 500 still carries `nosniff`); malformed JSON is a 400 before dispatch; bodies are capped at node parity (1 MiB).
- **Verified good:** path-escape containment in both write seams (`packages/sites/src/write.ts:28-36`, `packages/deploy/src/ship.ts:93-99`) and version-segment validation incl. the pointer itself (`packages/deploy/src/release.ts:80-88, 178`); client IP comes only from `cf-connecting-ip`, which the edge sets and clients cannot forge (`fetch-handler.ts:291`); query flattening is last-wins on both tiers (`fetch-handler.ts:151-159`, `packages/runtime/src/request.ts:64-69`) ‚Äî no parameter-pollution divergence.

## Simplicity

- **P1 ‚Äî `packages/cloudflare/src/wrangler.ts:63-85` vs `examples/estate/wrangler.jsonc:2-28` ‚Äî the generator cannot generate the config it claims to.** The committed file says "Generated by `wranglerConfig`... then committed," but the generator emits only `name/main/compatibility_date/compatibility_flags/assets`; the deployed config additionally needs the six-entry Preact `alias` block and the react-dom shims ‚Äî hand-maintained, with nothing detecting drift (e.g. a `compatibility_date` bump or binding rename that never reaches the generator's tests). Recommendation: either grow `WranglerOptions` to cover what real deploys need (`alias`, `vars`, placement) and add a round-trip test that regenerates `examples/estate/wrangler.jsonc`, or delete the "generated by" claim and demote `wranglerConfig` to a scaffold helper.
- **P2 ‚Äî `packages/cloudflare/src/wrangler.ts:64` + `packages/deploy/src/plan.ts:67-84` ‚Äî "node" target conflated with "the Worker's dynamic zone."** `wranglerConfig` finds the dynamic zone via `kind === "node"`, but that target asserts `run: "volo serve"` and `needsDatabase: true` ‚Äî neither true of a Worker deploy. ADR 0001 anticipated a distinct `edge` adapter; the plan model never grew one. Recommendation: add a `worker`/`edge` target kind (or a render-mode discriminant) so the plan describes the deploy that actually happens.
- **P2 ‚Äî `packages/deploy/src/plan.ts:94-99` ‚Äî the routing manifest is dead weight on the headline target.** `plan.routing` is documented as "the single source an edge/CDN reads," but the Cloudflare path routes implicitly via assets-first `withAssets`; only the node front door consumes it. Two routing truths. Recommendation: feed `withAssets` (or a dynamic-prefix skip list) from `plan.routing`, making the manifest load-bearing on both tiers.
- **P2 ‚Äî `packages/cloudflare/src/index.ts:6-14` ‚Äî the package's own usage example imports `dispatchSites` from `@volo/runtime` into a Worker**, which drags the node-flavored runtime (node:fs static reader, node:http server) toward an edge bundle; the real worker (`examples/estate/worker.ts:63`) correctly fronts `app.handle` + the ASSETS binding instead. Recommendation: update the doc example to the pattern the estate actually deploys.
- **Verified good:** every IO surface is a small injected seam (`ShipDeps`, `ReleaseStore`, `OutputSink`, `AssetFetcher`, structural `EdgeRequestTracer` with no dependency on `@volo/observability`); the three packages total ~900 lines of source with 100% coverage and zero external deps.

## Durability

- **P1 ‚Äî `packages/deploy/src/ship.ts:27-37, 86-88` ‚Äî the ship seam is string-typed and reads UTF-8, so binary assets are corrupted on publish.** `read` returns `Promise<string>` via `readFile(..., "utf8")` and `put` takes `contents: string`, while `contentTypeFor` happily advertises `png/jpg/woff2/ico` (`packages/deploy/src/content-type.ts:20-28`). Today's prerenderer only emits HTML strings, so the bug is latent ‚Äî but the first image or font that flows through `shipStatic`/`shipRelease` ships garbage bytes. Recommendation: retype the seam to `Uint8Array` (and `OutputSink` with it), keeping `string` as a convenience overload.
- **P1 ‚Äî `@volo/deploy` does not upload, and the real Cloudflare deploy never touches it.** The only concrete `ShipDeps`/`ReleaseStore` are local-disk (`ship.ts:82-105`, `release.ts:169-207`); the documented Cloudflare runbook is `build.ts` ‚Üí `wrangler deploy` (`examples/estate/build.ts:9-10, 34`), where wrangler itself uploads `out/marketing` ‚Äî so `shipRelease`'s carefully-ordered stage‚Üíverify‚Üíflip and `rollback` are unreachable on the headline target. Answering the review question directly: **the uploader is injected, and only a local filesystem default exists.** Recommendation: ship one real remote store (R2 or the Workers Assets API) implementing `ReleaseStore`, so versioned release/rollback applies to the edge deploy rather than only to self-hosted node.
- **P2 ‚Äî `packages/cloudflare/src/fetch-handler.ts` ‚Äî no per-request deadline on the edge.** The node server times out handlers and answers 503 (`statusForError` maps `RUNTIME_HANDLER_TIMEOUT`, `harden.ts:132`), but nothing on the edge enforces a deadline; a hung dispatch holds the request until the platform kills the isolate. Recommendation: optional `timeoutMs` racing dispatch against the context's signal, answering the same coded 503.
- **P2 ‚Äî `packages/sites/src/types.ts:50-71` ‚Äî prerender is string-bodied only.** `RenderResponse.body: string` means a streamed or bytes-bodied route (an og-image endpoint, a large feed) cannot live in a static zone; it silently doesn't fit the type rather than failing with a coded error. Acceptable v1 limit; document it and widen alongside the ship seam fix.
- **P2 ‚Äî `packages/deploy/src/release.ts:199-205` ‚Äî no release pruning.** `listReleases` grows forever; every release's full tree stays on disk with no GC. Recommendation: a `prune(keep: n)` that refuses to delete `current`.
- **Verified good:** all-or-nothing build gate ‚Äî a single non-2xx page fails the whole build before any write (`packages/sites/src/build.ts:77-99`); release ordering is stage ‚Üí health-gate ‚Üí atomic pointer flip, with a failed gate leaving the staged tree for inspection (`release.ts:109-131`, ordering proven at `deploy.test.ts:286-348`); the symlink cutover is rename-atomic with stale-staging cleanup (`release.ts:177-189`, test `deploy.test.ts:443`); rollback refuses unknown versions (`release.ts:140-158`).

## Observability

- **P1 ‚Äî `packages/cloudflare/src/fetch-handler.ts:123-128, 357-360` + `examples/estate/worker.ts:71` ‚Äî the adapter promises `waitUntil`-flushed tracing but never plumbs `ExecutionContext`.** The `tracer` option's own doc says to "pair with an `OtlpHttpExporter` flushed through the Worker's `waitUntil`," yet `toFetchHandler` returns `(request) => Promise<Response>` and the estate worker's `fetch(request, env)` drops `ctx` ‚Äî so there is no way to schedule a post-response export; spans flushed after `return` run in a frozen isolate and are silently lost. Recommendation: accept `(request, ctx?)` (or an `EdgeOptions.afterResponse` hook receiving `waitUntil`), and wire the estate worker as the reference OTLP-on-Workers example.
- **P2 ‚Äî `packages/cloudflare/src/fetch-handler.ts:94-96` ‚Äî default edge access log is an unstructured `console.log` line.** Workers Logs/Logpush ingest structured JSON far better; the node tier's entry shape exists, but the default rendering throws the structure away. Recommendation: default to `console.log(JSON.stringify(entry))` on the edge.
- **P2 ‚Äî `examples/estate/worker.ts:70-79` ‚Äî the flagship worker wires no `tracer` and no custom `logRequest`,** so the deployed example demonstrates none of the observability surface the adapter grew (spans, request-id-stitched logs). Recommendation: once `waitUntil` lands, ship the estate worker with the OTLP exporter wired as the canonical pattern.
- **Verified good:** one access entry per request with method/path/status/latency/request-id mirroring the node `AccessEntry` (`fetch-handler.ts:58-70, 408-414`); spans carry status code + request id and flag only 5xx as error (`fetch-handler.ts:416-424`, test `cloudflare.test.ts:168-224`); 4xx client errors are deliberately not logged as server failures (`fetch-handler.ts:333-339`, test `cloudflare.test.ts:379-391`); deploy errors all carry stable codes (`packages/deploy/src/errors.ts:12-22`).

## Top launch-blockers

1. **P0 ‚Äî Silent fallback to a committed session-signing secret on the deployed Worker** (`examples/estate/src/edge.ts:238-240`). One forgotten `wrangler secret put` and every session on the launch example is forgeable from public source. Fail closed in production; one-line fix plus a test.

(That is the only P0. The nearest P1s ‚Äî the edge app's missing `secureStack` and the `?as=` impersonation route ‚Äî compound it and should land in the same hardening pass.)

## Recommended plan items

1. **Fail closed on the edge secret** ‚Äî throw when `env.SESSION_SECRET` is absent outside an explicit demo mode; e2e test that an unset secret refuses to serve. (P0, hours)
2. **Bring the edge twin to node-twin posture** ‚Äî mount `fromRequestMiddleware(secureStack({ originCheck: {} }))` + per-isolate rate limit in `buildEdgeApp`; replace/gate the `?as=` sign-in with `@volo/identity`; add a cross-origin-POST-refused e2e. (P1)
3. **Plumb `ExecutionContext` through `toFetchHandler`** and wire the estate worker with the OTLP exporter flushed via `waitUntil` ‚Äî make the flagship the observability reference. (P1)
4. **Fix the ship seam to bytes** (`Uint8Array` through `ShipDeps`/`OutputSink`) before any non-HTML asset flows through deploy. (P1)
5. **Make `wranglerConfig` honest** ‚Äî add `alias`/`vars` support and a round-trip test regenerating `examples/estate/wrangler.jsonc`, or drop the "generated" claim. (P1)
6. **One real remote `ReleaseStore`** (R2 or Workers Assets API) so versioned release/rollback applies to the Cloudflare target, not just self-hosted node. (P1, the substance of "deploy" for v1)
7. Post-launch: edge ETag via Web Crypto; hashed island bundles + cache policy on `PublishFile`; a `worker` target kind in the deploy plan and routing-manifest-driven `withAssets`; release pruning; structured default edge access log. (P2s)
