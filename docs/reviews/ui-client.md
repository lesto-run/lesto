# UI & Client Pipeline ‚Äî Architecture Review

Reviewed 2026-06-11 against the working tree (HEAD `be90c34`). Every claim below was verified in source; bundle sizes were measured with `bun build --target browser --minify` against the actual package sources.

## Domain summary

The island architecture itself is genuinely good: one audited serialization seam (script-context escaping fixed ‚Äî the prior P0 is closed), a single `islandMount` authoring the wire shape for both emission paths, per-request memoized data resolution with a 0-RTT inline tier (ADR 0012), parse-time data priming, real per-island code splitting for `visible` islands, mount resilience that contains one broken island, and a streaming renderer with a hard deadline. The pipeline is wired end-to-end (CLI ‚Üí `@keel/assets` ‚Üí synthesized entry ‚Üí `hydrateDocumentIslands`; blog ships a live data-bound island). But the framework's core bet ‚Äî "feels like a hand-tuned SPA" ‚Äî is currently lost at the last mile: the synthesized client entry imports the `@keel/ui` barrel, which statically drags `react-dom/server` and the whole server render machinery into the browser bundle (**measured: 118 KB gzip, vs 60 KB with the barrel bypassed ‚Äî the server renderer nearly doubles the client**), and the CLI hardcodes `dialect: "react"`, so the ~10 KB Preact story (ADR 0007) is unreachable through `keel dev`/`keel build`. Ship-shape engine, mis-shipped bundle. Fix the barrel split and the dialect key and the headline claim becomes true by default.

## Performance

- **P0** ‚Äî `packages/assets/src/synthesize.ts:48` + `packages/ui/src/index.ts:25-39` + `packages/ui/src/render.tsx:26`, `packages/ui/src/stream.tsx:42`: the synthesized client entry does `import { Registry } from "@keel/ui"`, and the barrel statically imports `react-dom/server` (render + stream modules). Measured with the real sources: barrel entry = **381 KB raw / 118 KB gzip**; identical entry importing `registry.ts`/`hydrate.tsx` directly = **189 KB raw / 60 KB gzip**. Every app built by the default pipeline ships ~58 KB gzip of dead server renderer to every visitor. The preact shims (`packages/assets/src/preact-alias.ts:29-34`) exist *because* of this drag ‚Äî they paper over it for the preact dialect only; the default react dialect ships it silently. Recommendation: split the barrel (a `@keel/ui/server` subpath for render/stream/resolver, keep `Registry`/island types isomorphic) and make `synthesizeEntry` import only isomorphic modules. This is acknowledged as "chief-architect review 2a" in `define-island.tsx:92-94` but not done.
- **P1** ‚Äî `packages/cli/src/bin.ts:67-78`: `dialect: "react"` is hardcoded (`// "react" for now ‚Äî the ui.dialect config key is Increment 2/3 scope`). ADR 0011's one-key `ui: { dialect }` does not exist, so the ~118 KB ‚Üí ~10 KB gap (ADR 0007) cannot be closed by an app using the CLI; only estate's bespoke worker gets Preact. Recommendation: land `keel.config` `ui.dialect` driving client alias + server renderer atomically (the "matched pair", ADR 0008) before 1.0.
- **P1** ‚Äî `packages/web/src/render-page.tsx:213` + `packages/ui/src/stream.tsx:42,136`: the canonical `.page` path always streams via React's `renderToReadableStream`; the `ServerRenderer` seam (ADR 0008) only covers the buffered `renderPageMarkup` (`packages/ui/src/render.tsx:157-173`). There is no Preact streaming adapter and no way to inject one, so an `ssr: true` island under the preact dialect is impossible on the framework's primary path ‚Äî the dialect key (above) cannot be honest until this seam reaches the stream. Recommendation: extend the renderer seam to `renderPageResponse` (preact's `renderToReadableStream` analogue, or buffered fallback under preact).
- **P2** ‚Äî `packages/web/src/keel.ts:237-239` + `packages/web/src/render-page.tsx:226-228`: registering *one* `private` data source stamps **every** page in the app `Cache-Control: private, no-store` ‚Äî including island-free marketing pages. Documented as deliberate (streaming makes per-page detection hard), but it is an app-wide caching cliff. Recommendation: per-route opt-out, or track at registration which pages' island graphs bind private sources.
- **P2** ‚Äî `packages/assets/src/build-client.ts:83-89, 137-149`: the build reports written paths only. ADR 0011 promises "the build narrates what it decided (dialect, per-island sizes, chunks, why-split)" and a size heuristic; neither exists, so the P0 above was invisible. Recommendation: emit per-artifact gzip sizes + a configurable budget that fails the build.
- Verified good: parse-time primer is idempotent and parallel with `client.js` (`packages/ui/src/data.ts:165-183`); `visible` islands get true byte deferral via dynamic import (`packages/assets/src/synthesize.ts:55-64`); chunk+data fetched in parallel, never waterfalled (`packages/ui/src/hydrate.tsx:416-428`); the marker rule (`renderToString` iff any `ssr` island) is mechanical (`packages/ui/src/render.tsx:168-172`); resolver memoizes one loader run per source per request (`packages/ui/src/data-resolve.tsx:79-102`).

## Security

- **Verified fixed (prior P0)** ‚Äî island `<script>` JSON is properly escaped: `serializeScriptJson` escapes `<`, `>`, `&`, U+2028/29 to `\uXXXX` (`packages/ui/src/serialize.ts:131-143`), emission is inert `type="application/json"` (`packages/ui/src/define-island.tsx:203-210`; `examples/estate/src/document.ts:118`), and the client revives with `JSON.parse(textContent)`. Both emission paths share the seam.
- **P1** ‚Äî `packages/ui-kit/src/components.tsx:239-244` and `packages/forms/src/components.tsx:48-53`: `Button.href` and `Form.action` are interpolated into `<a href>` / `<form action>` with no scheme check. The threat model here is the package's own premise ‚Äî AI-generated trees (a prompt-injected model, or a stored tree from an admin) can emit `href: "javascript:..."`, which React 19 renders with only a console warning. The "vetted vocabulary" has a scheme hole in its two link-shaped props. Recommendation: allowlist `https:`/`http:`/relative (and `mailto:` if desired) in both renders; fall back to rendering a `<button>`/dropping `action`.
- **P2** ‚Äî inline primer is *executable* inline JS (`packages/ui/src/define-island.tsx:211-213`, `packages/ui/src/data.ts:165-183`) with no nonce/hash seam, and `renderPageStream`'s `bootstrapScriptContent` is likewise nonce-less. Under the strict CSP the serialize-module doc itself recommends ("compatible with a strict, nonce-based CSP", `serialize.ts:108-110`), the primer is blocked and bound islands silently fall back to the post-hydration fetch. Recommendation: accept an optional `nonce` through `RenderPageOptions` ‚Üí primer/mount-script/bootstrap emission.
- **P2** ‚Äî `packages/web/src/keel.ts:241-245`: `/__keel/data/<name>` routes inherit only middleware declared *before* `.data()` ‚Äî auth declared after registration silently doesn't guard the loader. Order-sensitivity is documented for `.use` generally, but a per-user data endpoint is the worst place for it. Recommendation: register data routes at dispatch time (compose the full final chain) or lint the ordering.
- Verified good: source names charset-locked before they touch a route or the primer (`packages/ui/src/data.ts:67,93-99`); `private`-by-default scope drives `no-store` on the data route (`packages/web/src/keel.ts:227-229`); props proven JSON-pure before the wire (`packages/ui/src/serialize.ts:75-91`); primer/bind fetches are `credentials: "same-origin"`; ui-kit guards numeric props from CSS injection (`packages/ui-kit/src/components.tsx:323-325`); generated trees are re-validated server-side (`packages/ui-generate/src/generate.ts:86`).

## Simplicity

- **P1** ‚Äî two complete, parallel island systems: the Registry/UiNode path (`renderPage` ‚Üí page-wide manifest ‚Üí `serializeManifest` ‚Üí `hydrateIslands`, `packages/ui/src/render.tsx:113-121` + `hydrate.tsx:342`) and the `.page`/`defineIsland` path (co-located mount scripts ‚Üí `hydrateDocumentIslands`, `define-island.tsx` + `hydrate.tsx:517`). `islandMount` (`packages/ui/src/mount.ts:37`) heroically keeps the wire byte-identical, but every island invariant must now be enforced (and audited) in two emission sites and two client entry points ‚Äî the `UI_ISLAND_SSR_DATA_UNRESOLVED` rule already "drifted onto only the defineIsland path" once (`mount.ts:82-90`). ADR 0011 Increment 2 promises convergence. Recommendation: after estate converges, demote the Registry island path to the content/DB-driven niche or delete it; keep one client entry point.
- **P2** ‚Äî `packages/ui/src/props.ts` is a third validation dialect (PropSpec) beside Zod-at-the-boundary (ADR 0005, used by `.page` `params`) and forms' own `validateSubmission`. Three bespoke validators for one framework. Recommendation: post-1.0, fold PropSpec onto Zod (it already feeds JSON Schema generation, which Zod 4 does natively).
- **P2** ‚Äî `packages/ui/src/island.ts:117-125`: `defineClient`'s `data` typing is deliberately erased while `defineIsland` carries the typed `DataSource<P[K]>` binding ‚Äî acknowledged, fine, but it is one more asymmetry the convergence above should erase rather than document.
- Verified good: the injectable-seam discipline (`mount`/`observe`/`onRecoverableError`/`ServerRenderer`/`BuildClientDeps`) keeps every runtime-varying thing testable without a browser or bundler; `@keel/forms` and `@keel/ui-kit` are small, dependency-honest packages.

## Durability

- **P1** ‚Äî `packages/assets/src/build-client.ts:127-135`: the stale-chunk sweep deletes the *previous* build's `chunk-*.js` **before** writing the new entry and chunks. Two failure windows: (a) a live `keel dev` server (rebuilds fire on watch, `packages/cli/src/run.ts:453`) serves the old entry ‚Äî still on disk ‚Äî whose dynamic imports now 404, so every `visible` island on an open page dies on first scroll; (b) a crash between sweep and write leaves an out-dir with an entry and no chunks. Recommendation: write new artifacts first, then sweep anything not in the new set (the hashed names make this trivially safe); for deploys, keep N generations so CDN-cached documents in flight keep working.
- **P2** ‚Äî `packages/assets/src/bun.ts:50-58`: `readIsland` dynamic-imports each app island into the build process and dereferences `module.default.island` with no shape check ‚Äî a file under `app/islands/` without a `defineIsland` default export crashes the build with a raw `TypeError`, not a coded `AssetsError` naming the file. It also executes arbitrary app top-level code in the builder (a browser-only global access throws here). Recommendation: validate the module shape, wrap in `ASSETS_BAD_ISLAND_MODULE` with the path.
- **P2** ‚Äî `packages/ui/src/registry.ts:32-37, 49-56`: `define`/`defineClient` silently delete the same name from the *other* namespace (last-write-wins across server/client). A typo'd collision converts a server component into an island (or vice versa) with no signal. Recommendation: warn or throw on cross-namespace shadowing unless explicit.
- Verified good: render never throws ‚Äî unknown/malformed/throwing nodes degrade and are reported (`packages/ui/src/render.tsx:176-291`); one island's mount failure cannot dark out the rest, including deploy-skew unknown components (`packages/ui/src/hydrate.tsx:433-454`); bind resolution races a 10 s deadline mirroring the server's (`hydrate.tsx:92,136-156`); streaming has a hard `UI_STREAM_TIMEOUT` deadline and the buffered exit refuses half-rendered documents (`UI_STREAM_INCOMPLETE`, `stream.tsx:334-341`); corrupt mount scripts are skipped, not fatal (`hydrate.tsx:525-538`); bfcache helper structurally forbids `unload` (`packages/ui/src/bfcache.ts`).

## Observability

- **P1** ‚Äî production islands fail silently: the synthesized entry discards the `HydrationResult` (`packages/assets/src/synthesize.ts:78` emits a bare `hydrateDocumentIslands(registry);`) and the default sinks are `console.error` in the visitor's browser (`packages/ui/src/hydrate.tsx:277-284`). The framework that promises "loud-when-wrong" (ADR 0011) has no way for an operator to learn that 30 % of mounts are failing after a deploy skew. Recommendation: synthesize an entry that forwards `failed`/`missing` + `onMountError`/`onRecoverableError` to a tiny beacon (e.g. POST `/__keel/client-errors`, sampled), wired to the kernel's existing OTLP tracing; dev mode gets the ADR-promised overlay.
- **P2** ‚Äî `packages/ui/src/render.tsx:266-270, 286-290`: `catch { errors.push({ path, type: "render_threw" }) }` discards the thrown error entirely ‚Äî a throwing component yields only a path and the string `render_threw`, with the message, stack, and any coded `UiError` (e.g. `UI_ISLAND_PROPS_NOT_SERIALIZABLE`, deliberately routed here) unrecoverable. Recommendation: carry `detail`/`cause` on `RenderError`.
- **P2** ‚Äî prop validation errors vanish on the render path: `render.tsx:206` and `mount.ts:43` destructure `{ props }` and drop `errors`, so a missing required prop renders silently unless the caller separately ran `validateTree`. Recommendation: push `validateProps` errors into the walk's `errors` as `invalid_props` diagnostics.
- **P2** ‚Äî ADR 0011's status block is stale against the code (claims "render-page.tsx does not yet emit the head module tag", "`@keel/assets` is not yet CLI-invoked", "examples/blog has no island" ‚Äî all three are now built: `render-page.tsx:202-204`, `cli/bin.ts:73`, `examples/blog/app/islands/reactions.tsx`). Given the team's own "verify 'built' claims" rule, doc drift in the flagship ADR is an observability bug. Recommendation: update the status block.
- Verified good: every refusal across all five packages is a coded error (`UiError`/`AssetsError`/`GenerateError`); stream errors flow through an injectable sink with `componentStack`; `UI_STREAM_TIMEOUT` is distinguishable from genuine render errors by code.

## Top launch-blockers

1. **P0 ‚Äî the default client bundle ships the server renderer.** `@keel/assets`' synthesized entry imports the `@keel/ui` barrel, dragging `react-dom/server` into the browser: measured 118 KB gzip vs 60 KB without the barrel ‚Äî and the CLI's hardcoded `dialect: "react"` (`packages/cli/src/bin.ts:78`) means no app can reach the ~10 KB Preact bundle the framework's pitch is built on. Split the barrel (`@keel/ui/server` subpath), point `synthesizeEntry` at isomorphic modules only, then land the `ui.dialect` config key. This single item is the difference between the headline claim being true and false by default.

## Recommended plan items

1. **Split the `@keel/ui` barrel** into isomorphic core (`Registry`, island types, `defineIsland`, data tokens) and `@keel/ui/server` (render/stream/resolver/metadata/resources); re-point `synthesizeEntry`; delete the two react-dom shims this makes unnecessary; add a bundle-size assertion test (‚â§ 65 KB gzip react, ‚â§ 15 KB preact) so the regression can never silently return.
2. **Land `ui: { dialect }` in keel.config** driving client alias + server renderer as the matched pair (ADR 0008/0011), including a streaming `ServerRenderer` seam in `renderPageResponse` (or a documented buffered fallback under preact); make `create-keel` scaffold `dialect: "preact"` (Increment 3).
3. **Make the chunk sweep write-then-sweep** (and keep one previous generation in production builds) in `buildClient`.
4. **Scheme-guard `Button.href` and `Form.action`** (allowlist http/https/relative) ‚Äî closes the one real XSS vector left in the AI-tree path.
5. **Ship the client error beacon**: synthesized entry forwards `HydrationResult` + mount/recoverable errors to a sampled `/__keel/client-errors` endpoint feeding the existing OTLP pipeline; dev overlay for the same sinks.
6. **Converge the island paths** (ADR 0011 Increment 2): estate onto `defineIsland`/`hydrateDocumentIslands`, then demote or delete the Registry-manifest island path and `hydrateIslands`' public manifest form.
7. **Diagnostics polish** (small, batchable): carry the thrown error on `render_threw`; surface `validateProps` errors in the render walk; coded error for malformed island modules in `@keel/assets`; warn on Registry cross-namespace shadowing; emit per-artifact sizes from `buildClient`; refresh ADR 0011's status block.
8. **Optional nonce plumbing** for the primer/bootstrap inline scripts, completing the strict-CSP story the serializer already designed for.
