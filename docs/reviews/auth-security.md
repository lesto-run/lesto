# Auth & Security — Architecture Review

## Domain summary

The primitives are genuinely good — better than typical for a young framework. Timing-safe comparisons everywhere, 256-bit tokens, session-bound CSRF with the null-byte delimiter trick, fail-open scrypt verification explicitly defended against, wildcard+credentials CORS rejected at config time, default-deny across authz/rbac/flags, `__Host-` cookies, enumeration defenses with CPU equalization, and a reset token made single-use by mixing the password hash into the HMAC secret. All 219 tests across the eight packages pass (verified by running them: auth 41, identity 48, csrf 35, cors 20, ratelimit 34, authz 18, rbac 14, flags 9). ADR 0013's durable-store claims are real in code. The prior findings C1/H3/M4 are fixed; H2 and #6 are fixed **only on the Node path** — the deployed Cloudflare edge variant still mints sessions with zero credential verification behind a hardcoded fallback secret, which is the one launch-blocker. The systemic weaknesses are upgrade-path rigidity (scrypt params frozen and unencoded in the hash format), production defaults that silently fall back to in-memory stores, no brute-force story for the batteries-included login, and a security battery that emits no audit events at all.

## Performance

- **P1 — `packages/auth/src/password.ts:28,61` — synchronous scrypt on the request path.** `scryptSync` blocks the event loop for every register/login/reset (~tens of ms each at current params; more once params are raised per the Security finding). Under a login flood this is a DoS amplifier: one CPU-bound call head-of-line-blocks every concurrent request on the node. `packages/identity/src/identity.ts:95` also runs it at module load (`DUMMY_HASH`). Every caller is already async (ADR 0013 made the whole chain async), so switch to promisified `crypto.scrypt` with an explicit `maxmem`; keep the sync form only if a sync seam truly needs it.
- **P2 — `packages/ratelimit/src/sql-store.ts:120-160` — one transaction + locked read per request.** `sqlRateLimitStore` runs `BEGIN`/`SELECT … FOR UPDATE`/write/`COMMIT` per check. Correct (and well-reasoned), but as a *global* middleware it doubles every request's DB round-trips. Document that the SQL store belongs on hot auth endpoints, not the whole app, or front it with a short-window in-memory cache; on Workers the right substrate is a Durable Object, which ADR 0013 defers.
- **P2 — `packages/rbac/src/permissions.ts:54-66` — `can()` re-resolves the full inheritance graph per call** (fresh `Set`s, transitive walk). Fine at demo scale; memoize resolved grants per role if rbac survives the consolidation below.

## Security

Verified clean before findings: timing-safe compares with length pre-guards (`packages/auth/src/password.ts:63`, `packages/auth/src/signed-sessions.ts:69-76`, `packages/csrf/src/token.ts:69-73`); fail-open scrypt verify explicitly blocked by fixed-width checks (`password.ts:52-58`); 256-bit session tokens (`packages/auth/src/token.ts:4`); CSRF session binding with `\0` delimiter so `(nonce, sessionId)` can't be spliced (`token.ts:24-27`); `originCheck` deliberately blind to Content-Type (immune to the SvelteKit/Astro/Hono CVE class) and fail-closed on missing signals (`packages/csrf/src/origin.ts:87-110`); CORS wildcard+credentials throws (`packages/cors/src/cors.ts:115-123`); deny-by-default in policy (`packages/authz/src/policy.ts:96-108`), rbac, and flags (off-by-default with `Object.hasOwn` prototype guard, `packages/flags/src/flags.ts:62-66`); `__Host-` + `HttpOnly` + `Secure` + `SameSite=Lax` cookies (`packages/identity/src/cookies.ts:30-37`); enumeration defenses with CPU equalization on register/reset/login (`packages/identity/src/identity.ts:263-266,340-343,373-376`); single-use reset via password-hash-bound secret (`packages/identity/src/tokens.ts:52`); no secret, token, or password is ever logged anywhere in the eight packages.

- **P0 — `examples/estate/src/edge.ts:201-202,239` — the deployed edge path still has credential-less auth and a hardcoded fallback secret.** Prior finding #6 (`?as=` impersonation) was fixed only for Node: the edge twin's `/mls/api/sign-in` mints a signed session for `USERS.get(c.query("as") ?? "jade")` with zero credential verification, and `edgeSecret()` falls back to the literal `"estate-demo-edge-secret"` — a public string under which anyone can forge a session token for any user on a deployment that forgets `SESSION_SECRET`. The edge app also mounts no `originCheck` and no rate limit (prior H2 fixed only via `app.ts:39` on Node). If the Cloudflare estate deployment is the launch artifact, this is launch-blocking: either port the Identity flow to the edge (D1/KV seeding, per ADR 0003 Phase 2), or hard-fail boot when `SESSION_SECRET` is absent and gate `/mls/api/sign-in` behind an explicit `DEMO_AUTH=1` binding.
- **P1 — `packages/auth/src/password.ts:28-30,61` — scrypt parameters are frozen, below current guidance, and not encoded in the hash.** `scryptSync(password, salt, 64)` uses Node defaults (N=2^14, r=8, p=1); OWASP guidance for scrypt is N=2^17. Worse, the stored format is `scrypt$salt$hash` with **no cost parameters**, so parameters can never be raised without invalidating every stored credential. Adopt `scrypt$N$r$p$salt$hash`, verify against the stored params, and rehash-on-successful-login when params are stale. This is a one-way door: every password hashed before the format change compounds the migration.
- **P1 — `packages/auth/src/signed-sessions.ts:103`, `packages/csrf/src/token.ts:36`, `packages/identity/src/identity.ts` — no secret-strength validation anywhere.** `SignedSessions`, `csrf`, and `createIdentity` accept any string, including `""`. An empty or short secret makes every session and CSRF token forgeable, silently. Estate demonstrates exactly this failure mode (`examples/estate/src/identity.ts:128` falls back to `"estate-demo-identity-secret"`). Throw a coded error at construction for secrets under 32 bytes.
- **P1 — `packages/auth/src/sql-session-store.ts:34` — session tokens stored in plaintext** (`token TEXT PRIMARY KEY`). A DB snapshot, replica, or backup leak yields live, immediately-usable session credentials for every user. Store `sha256(token)` as the key — `find`/`delete` hash before lookup; one line each, no schema change beyond semantics.
- **P1 — `packages/runtime/src/trust-proxy.ts:98` — `trustProxy: true` believes the left-most `X-Forwarded-For` entry**, which the *client* writes. Behind one proxy, a client sending `X-Forwarded-For: 6.6.6.6` produces chain `[6.6.6.6, realIP]` and `chain[0]` is the forgery — so the most common config hands the attacker the rate-limit key (`packages/ratelimit/src/middleware.ts:150` keys on this `ip`) and the audit identity: a full per-client rate-limit bypass. The numeric-hop mode is correct. Make `true` mean "one trusted hop" (take `chain[length-1]`), or deprecate `true` in favor of the numeric form.
- **P1 — no brute-force protection on the batteries-included login.** `identity.login` has no attempt throttling or lockout; `secureStack` rate limiting is IP-keyed only and opt-in; estate mounts no `rateLimit` at all (`examples/estate/src/app.ts:39`). Per-IP buckets don't stop credential stuffing across a botnet. Ship a documented recipe (or default) keying a `RateLimiter` on `login:<normalizedEmail>` inside identity, with the SQL store so it holds fleet-wide.
- **P1 — `packages/identity/src/identity.ts:228,430-431` — sessions survive password reset by default.** `revokeUserSessions` is an optional hook, and the default `MemorySessionStore` can't iterate by user anyway. The compromised-account flow — attacker holds a session, victim resets password — leaves the attacker logged in for up to 7 days. Now that `sqlSessionStore.deleteByUserId` exists (ADR 0013), wire it by default whenever the store supports it, and call it from `resetPassword` unconditionally.
- **P2 — `packages/csrf/src/origin.ts:100` — `originCheck` treats `Sec-Fetch-Site: same-site` as safe.** A compromised or attacker-controlled sibling subdomain (`evil.example.com` → `app.example.com`) is `same-site`, not `cross-site`, and passes. Add a `strict` option requiring `same-origin`/`none`.
- **P2 — `packages/identity/src/identity.ts:351` — `IDENTITY_EMAIL_NOT_VERIFIED` is a distinct login error**, leaking that the email is registered. Documented as a deliberate UX-over-leak trade (module header, line 31); keep it documented, but the docs should tell apps with strict enumeration requirements to map both codes to one message.
- **P2 — `packages/cors/src/middleware.ts:43-45` — every `OPTIONS` is consumed as a preflight** (no `Access-Control-Request-Method` check), so a real OPTIONS route can never be reached through this middleware; and `packages/cors/src/cors.ts:150-152` sets `Vary: Origin` only when an origin was echoed — a denied response (no ACAO) is cacheable and a shared cache can then serve it to the allowed origin (and vice versa). Gate the preflight answer on `Access-Control-Request-Method`, and emit `Vary: Origin` whenever the policy is not `"*"`.

## Simplicity

- **P1 — two competing authorization packages.** `@keel/authz` (policy + guard, wired to `@keel/web`) and `@keel/rbac` (wildcards + inheritance, pure, wired to nothing) overlap almost completely; rbac has no middleware and no consumer. A user cannot tell which is canonical. Fold rbac's wildcard/inheritance matching into `definePolicy` (or mark `@keel/rbac` legacy) before 1.0 freezes both APIs.
- **P2 — the CSRF token battery has no delivery half.** `generateToken`/`verifyToken` and the middleware exist, but nothing mints a token into a cookie or template helper — every app must invent the plumbing (estate ducked it entirely by using `originCheck`). Either ship the issuance helper (a `csrfToken(c)` that sets the companion cookie) or stop describing it as double-submit: as built it is an HMAC session-bound token, which is fine, but the docs say "cookie + form field" (`packages/csrf/src/index.ts:4`) and no cookie path exists.
- **P2 — `examples/estate/src/edge.ts:42-58` re-implements `SESSION_COOKIE` and `readCookie`** instead of importing `@keel/identity`'s cookie module; the contract ("readers and writers travel through this module so the contract cannot drift", `cookies.ts:7-8`) is already drifting in the framework's own example.
- **P2 — the local `SqlDatabase`/`SqlStatement` seam is now declared in 10 packages** (`packages/auth/src/types.ts:60-75`, `packages/ratelimit/src/types.ts:55-75`, …). ADR 0013 explicitly accepts this; agreeing — but it belongs on the pre-1.0 consolidation list it names.

## Durability

ADR 0013 verified against code: both store interfaces are strictly async (`packages/auth/src/types.ts:40-46`, `packages/ratelimit/src/types.ts:27-37`); the rate-limit store is one atomic `update` verb with `FOR UPDATE` on Postgres, plain-INSERT birth race with exactly-one retry and a coded second-conflict refusal (`packages/ratelimit/src/sql-store.ts:165-190`); timestamps are `BIGINT` with `Number()` read coercion for node-postgres (`sql-session-store.ts:97`, `sql-store.ts:130-134`); estate's Node path dogfoods `sqlSessionStore` (`examples/estate/src/identity.ts:127`); `packages/integration/test/durable-stores.integration.test.ts` exists. The ADR's claims hold.

- **P1 — durable stores shipped, but nothing defaults to them.** `createIdentity` defaults to `MemorySessionStore` (`packages/identity/src/identity.ts:228`) and `rateLimit` defaults to `MemoryRateLimitStore` (`packages/ratelimit/src/middleware.ts:137-143`) — so the out-of-the-box production posture is still "every deploy logs everyone out, every node multiplies the rate limit." For a batteries-included framework the pit of success (ADR 0011's own framing) demands the kernel wire `sqlSessionStore`/`sqlRateLimitStore` when a `db` is present, or at minimum a loud production-boot warning mirroring `RATELIMIT_UNKNOWN_CLIENT`.
- **P2 — sweeps exist but nothing runs them.** `deleteExpired` (`sql-session-store.ts:107-111`) and `sweep` (`sql-store.ts:192-198`) are caller-cadence by design, but no recipe, cron battery, or kernel hook invokes them — never-again-presented rows accumulate forever. Pair with `@keel/queue`/scheduler when one lands; until then document the cadence in the identity README.
- **P2 — credential format is unversioned** (same root cause as the scrypt-params finding): `scrypt$salt$hash` carries no algorithm version, so an argon2id migration (or any param change) has no place to stand. The format fix above resolves both.

## Observability

- **P1 — the security battery emits zero security events.** `identity.login`/`resetPassword`/`verifyEmail` have no success/failure hook, no counter, no log — grep confirms the only observability call in all eight packages is the rate-limiter's `console.warn` (`packages/ratelimit/src/middleware.ts:96-103`). Without a `login_failed` signal you cannot build lockout, anomaly detection, or an audit trail, and an operator cannot see a credential-stuffing run in progress. Add an injectable `onEvent(event: IdentityEvent)` seam (coded events, no secrets in payloads) — the same closure-factory style as `IdentityMailer`.
- **P2 — denials are invisible across the middleware.** CSRF 403s (`packages/csrf/src/middleware.ts:120`, `origin.ts:71`), authz 403s (`packages/authz/src/guard.ts:55-59`), and rate-limit 429s return responses but surface no hook/metric; `onDeny` exists for authz (response shaping only). A uniform optional `onDenied(kind, c)` across the secure stack would make the launch dashboards (OTLP tracing landed 06-11) able to graph attack pressure.
- **Positive:** every package carries stable machine-readable error codes (`AUTH_INVALID_HASH`, `IDENTITY_*`, `CORS_WILDCARD_WITH_CREDENTIALS`, `RATELIMIT_STORE_CONFLICT`, `AUTHZ_UNKNOWN_ROLE`), and the warn-once unknown-client latch with its stable code (`middleware.ts:31-37`) is exactly the right shape — extend that pattern rather than inventing a new one.

## Top launch-blockers

1. **P0 — estate edge (the deployed Cloudflare artifact) has decorative auth**: credential-less `?as=` sign-in (`examples/estate/src/edge.ts:201-202`), hardcoded fallback signing secret (`edge.ts:239`), no origin check, no rate limit. Prior findings #6/H2 were closed on the Node path only. Fix before anything public points at the Worker: fail boot without `SESSION_SECRET`, fence the demo sign-in, mount `originCheck` on the edge app.

That is the only P0. The package layer itself has no launch-blocking hole — its P1s are upgrade-path and default-posture debts that compound with every stored hash and every production deploy, so they are pre-1.0, not pre-launch.

## Recommended plan items

1. **Fence the edge demo** (P0): boot-fail on missing `SESSION_SECRET`, gate `/mls/api/sign-in` behind an explicit demo binding, mount `originCheck` in `buildEdgeApp`. Small, ships today.
2. **Versioned password-hash format** (`scrypt$N$r$p$salt$hash`), raise defaults to N=2^17, rehash-on-login, switch to async `scrypt` with `maxmem`. Do this before any real users register — it is the one finding that gets strictly more expensive with time.
3. **Secret-strength guard**: throw a coded error for secrets < 32 bytes in `SignedSessions`, `csrf`, `createIdentity`.
4. **Hash session tokens at rest** in `sqlSessionStore` (sha256 key), and **revoke user sessions on password reset by default** when the store supports `deleteByUserId`.
5. **Fix `trustProxy: true` semantics** to peel exactly one hop (right-most), with a changelog note; keep numeric mode as-is.
6. **Default durable stores via the kernel**: when `createApp` has a `db`, wire `sqlSessionStore` + `sqlRateLimitStore` (+ schema install after migrate) automatically; warn loudly in production when falling back to memory.
7. **Login throttling recipe**: per-account (`login:<email>`) limiter inside identity over the SQL store; document the IP-keyed `secureStack` limiter as the outer layer, not the defense.
8. **Identity event seam** (`onEvent`) emitting coded `login_succeeded`/`login_failed`/`password_reset`/`email_verified` events; wire estate to the OTLP tracing that landed 06-11 as the dogfood.
9. **Consolidate authz/rbac**: fold wildcard + inheritance into `definePolicy` or mark `@keel/rbac` legacy; one authorization story before the API freezes.
10. **CSRF delivery helper + CORS preflight/Vary fixes + `originCheck` strict mode** — batch of small correctness items (P2s above) that close the remaining gaps without API churn.
