# Operability, API Surface & DX — Architecture Review

## Domain summary

The libraries are good; the product is not wired. Every package in this domain is small, DI-seamed, coded-error-disciplined, and tested at or near 100% — but the three claims this domain exists to back are all currently false in practice. (1) **Observability**: `@keel/observability` is a clean OTel-shaped tracer with a working OTLP/HTTP exporter, and both servers expose a structural `tracer` seam — but *nothing in the repo ever passes a tracer*: `keel serve`/`dev` construct `serve()` without one, no config knob exists, and neither tier has a flush lifecycle, so a real Keel deployment emits zero spans. (2) **Agent control plane**: `@keel/mcp`'s tools are real and tested, but `startMcpServer` has zero callers — no `keel mcp` command, no example — and the tools carry no authz, no read-only mode, and no audit trail. (3) **DX loop**: `create-keel` scaffolds a project that cannot install (`@keel/*@latest` against unpublished private packages), cannot run `keel` (no `@keel/cli` dependency), and would crash `keel dev` anyway (no `keel.sites.ts`); the package's own e2e test never installs or boots its output, which is exactly the gap that let all three breaks ship. The webhook battery is the strongest piece — queue-backed retry, secret-by-reference, a well-tested SSRF guard — but the guard is bypassable by a plain HTTP redirect, defeating its documented invariant.

## Performance

- **P2** — `packages/observability/src/otlp.ts:119,139-141` — `OtlpHttpExporter.buffer` is unbounded: `export()` pushes forever and only `flush()` drains. Wired without a flush cadence (which is the current state — see Observability), it is a slow memory leak. *Recommendation: cap the buffer (drop-oldest + `onError` count) and/or add an interval-flush option.*
- **P2** — `packages/mcp/src/tools.ts:205,220,242` — every content write tool re-runs `hydrateRuntime(db)`, a full store rehydration per mutation. An agent doing a 50-entry import pays O(n²). *Recommendation: batch tool or post-write incremental hydration.*
- **P2** — `packages/webhooks/src/webhooks.ts:234,316` — each delivery resolves DNS twice (guard, then fetch's own resolution). Negligible cost, but fixing the TOCTOU below (pin the resolved IP) removes it for free.
- Good: the tracer-off default is genuinely zero-overhead (`packages/runtime/src/server.ts:815-817` — optional chain, no allocation); both tiers bound body reads while streaming (`packages/cloudflare/src/fetch-handler.ts:188-220`); the dev island watcher debounces rebuilds (`packages/cli/src/bin.ts:86-101`).

## Security

- **P0** — `packages/webhooks/src/webhooks.ts:296-316,66-69` — **the SSRF guard is bypassed by a redirect.** `deliver()` guards `payload.url`, then calls `fetchFn` — defaulting to global `fetch`, whose default is `redirect: "follow"`, and the `FetchLike` seam (`webhooks.ts:66-69`) cannot even express a redirect policy. A receiver answering `302 Location: http://169.254.169.254/...` walks straight past the guard the module's header comment (`webhooks.ts:24-27`) promises. *Recommendation: pass `redirect: "manual"`, treat 3xx as delivery failure (or re-guard each hop); widen `FetchLike` to carry the redirect option.*
- **P1** — `packages/webhooks/src/webhooks.ts:49-59,306-314` — **no replay protection.** The HMAC covers the body only; no timestamp or nonce is signed, so a captured delivery verifies forever. Stripe/GitHub-style signatures sign `timestamp.body` and `verify` enforces a tolerance window. *Recommendation: add an `x-keel-timestamp` header, sign `${timestamp}.${body}`, and give `verify` a `toleranceMs`.*
- **P1** — `packages/webhooks/src/webhooks.ts:234-247,316` — **DNS-rebinding TOCTOU**: the guard resolves the hostname, then `fetch` re-resolves it; a TTL-0 attacker answers public to the guard and private to the fetch. *Recommendation: connect to the guard-resolved IP (Host header preserved), or document the residual risk loudly.*
- **P1** — `packages/mcp/src/tools.ts:90-258`, `packages/mcp/src/server.ts:40-46` — **the MCP control plane has no authorization model at all.** Every tool — including `create/update/delete_content_entry` and `handle_request` (arbitrary route dispatch, any verb) — is unconditionally exposed to whatever client attaches; there is no read-only mode, no tool allowlist, no confirmation tier for destructive ops, and no rate limit. stdio's process boundary is the *only* guard, and nothing prevents this context from later being mounted on HTTP. *Recommendation: a `mode: "read-only" | "operator"` on `KeelMcpContext` gating the write/dispatch tools, plus per-tool `destructive` metadata.*
- **P1** — `packages/mcp/src/tools.ts:114-125` — `handle_request` forwards no headers (the options object carries only `query`/`body`), so agent-driven requests enter `app.handle` headerless. Header-based middleware (the scaffold's `originCheck` CSRF, any auth) sees a request with no identity — today that means agent POSTs are refused by secure apps (a usability wall), but for any app whose middleware fails open on a missing header it is an auth bypass channel. *Recommendation: either add an explicit, allowlisted `headers` input or document that `handle_request` is pre-middleware-hostile by design.*
- **P2** — `packages/openapi/src/openapi.ts:97-108` — `toOpenApi` documents every route with no include/exclude or visibility tag; the day it is served at `/openapi.json` it enumerates admin/internal surface. *Recommendation: per-route `internal` flag or a path-filter option before any serving story ships.*
- Good: `verify` is timing-safe with a length pre-check (`webhooks.ts:54-59`); secrets are persisted by reference only, resolved at delivery (`webhooks.ts:79,108-114` — and tested at `test/webhooks.test.ts:103`); the SSRF guard's IPv6 canonicalization and mapped-IPv4 handling are unusually thorough and pinned by ~20 tests (`test/webhooks.test.ts:209-326`).

## Simplicity

- **P0 (DX)** — `packages/create-keel/src/templates.ts:24,43-53` — **the scaffold→run loop is broken three independent ways**: (a) every `@keel/*` dependency is pinned `"latest"`, but every Keel package is `"private": true` and unpublished (e.g. `packages/db/package.json:6`, `packages/kernel/package.json:6`) — `bun install` fails on the first line of the README; (b) `scripts.dev` is `"keel dev"` but `@keel/cli` is not in the scaffold's dependencies (`templates.ts:34-53`), so the `keel` binary does not exist even after a hypothetical install; (c) `keel dev` unconditionally imports `keel.sites.ts` (`packages/cli/src/bin.ts:113-119`, `packages/cli/src/run.ts:441`) and the scaffold never writes one (`packages/create-keel/src/scaffold.ts:54-60`) — first run crashes with a module-not-found. *Recommendation: add `@keel/cli` to scaffold deps; make `loadSites` tolerate an absent `keel.sites.ts` (empty list); fix the version story (publish, or scaffold `file:`/tarball pins until then); and make create-keel's e2e actually run `bun install && keel dev` (see Durability).*
- **P1 (DX)** — `packages/cli/src/run.ts:129-147` + repo-wide grep — **no `keel mcp` and no `keel openapi` command.** `startMcpServer` (`packages/mcp/src/server.ts:24`) and `toOpenApi` (`packages/openapi/src/openapi.ts:91`) have zero callers outside their own packages. The "agent control plane" and "API surface" are libraries no operator can reach. *Recommendation: `keel mcp` (load `keel.app.ts`, wire context, start stdio) and `keel openapi [--out]` are each an afternoon given the existing seams.*
- **P2** — `packages/cli/src/flags.ts:48-54` — `parseStringFlag` happily takes the next flag as a value: `keel build --target --out dist` silently yields `target = "--out"`, then `CLI_UNKNOWN_TARGET` with a confusing payload. *Recommendation: reject values starting with `--`.*
- **P2** — `packages/cli/src/bin.ts:69-82` — client dialect is hardcoded `"react"` with a TODO, while the edge worker now defaults to preact (ADR 0009/0011 line); `keel dev` and a preact production deploy will hydrate different runtimes. *Recommendation: honor the planned `ui.dialect` config key before 1.0.*
- **P2** — in-house OTel-shaped tracing (`packages/observability/src/types.ts:1-8`) is a deliberate NIH bet. The current surface is admirably small; the risk is the next asks (context propagation, metrics, logs) re-growing OpenTelemetry one PR at a time. *Recommendation: adopt W3C `traceparent` verbatim instead of inventing a propagation format, and draw the line at traces — metrics/logs should be exporter-side.*
- Good: the whole domain follows one shape — pure decision core, injected seams, thin bin (`cli/src/run.ts:1-14`, `create-keel/src/scaffold.ts:1-7`, `mcp/src/server.ts:1-9`); every failure is a stable coded error; files are small and single-purpose.

## Durability

- **P1** — `packages/runtime/src/server.ts:559-585,649` + `packages/cloudflare/src/fetch-handler.ts:357-431` — **no trace flush lifecycle on either tier.** The node server's `close()` drains sockets but has no hook to flush an exporter; the edge `toFetchHandler` returns `(request) => Promise<Response>` with no `ExecutionContext`, so the `waitUntil` pairing its own docs recommend (`fetch-handler.ts:124-127`) has no seam to exist in. Even a determined operator who wires `OtlpHttpExporter` today buffers spans until process death. *Recommendation: `serve` accepts `onDrain`/flush hooks (interval + shutdown); `toFetchHandler` grows a `(request, ctx?)` arity and calls `ctx.waitUntil(exporter.flush())`.*
- **P1 (test gap)** — `packages/create-keel/test/bin.e2e.test.ts:44-54` — the e2e asserts files exist and exit codes; it never runs `bun install` or boots the output. All three P0 scaffold breaks are invisible to it. *Recommendation: a CI job that scaffolds into a temp dir, installs against workspace-linked packages, and curls `GET /posts` on the dev server.*
- **P2** — `packages/webhooks/src/webhooks.ts:296-304` — a `WEBHOOK_URL_BLOCKED` throw is retried by the queue like any transient failure; a permanently-private URL burns every attempt with backoff. *Recommendation: classify permanent vs transient failures (the queue needs a no-retry signal).*
- **P2** — `packages/observability/src/otlp.ts:144-164` — a failed flush drops the batch (documented demo-grade; no retry, no dead-letter). Acceptable for telemetry, but say so in the operator docs once a wire path exists.
- **P2** — `packages/cli/test/bin.e2e.test.ts:41-51` — the CLI's only e2e is `routes`; `serve`, `dev`, `deploy --release`, and `rollback` are never exercised as a spawned process. The covered core is excellent, but the bin wiring (dynamic imports, signal handlers, exit codes at `bin.ts:121-149`) is exactly where loops break.
- Good: webhook delivery rides `@keel/queue` retry/backoff with non-2xx-throws semantics (`webhooks.ts:316-327`); graceful shutdown drains in-flight requests on SIGTERM/SIGINT (`cli/src/bin.ts:136-143`); `keel serve` wires `/readyz` to a real `SELECT 1` probe, not a constant (`cli/src/run.ts:194-229`); scaffold refuses to clobber an existing directory (`create-keel/src/scaffold.ts:42-48`).

## Observability

- **P0** — `packages/cli/src/run.ts:220-224,467-471` + repo-wide grep — **the framework emits no spans on real requests.** Both servers expose a `tracer` seam (`packages/runtime/src/server.ts:223,815-894`; `packages/cloudflare/src/fetch-handler.ts:128,384-424`) and the span attribution code is correct and tested — but `keel serve` and `keel dev` build `serve()` without a tracer, there is no `keel.config`/env knob to enable one, no example wires `OtlpHttpExporter`, and (per Durability) nothing could flush it if they did. `@keel/observability` has zero non-test consumers. The in-house tracer is currently a library nobody calls. *Recommendation: the v1 increment is wiring, not code — `keel serve` reads `KEEL_OTLP_URL` (+ headers/service-name), constructs `Tracer` + `OtlpHttpExporter`, flushes on an interval and on drain; same envs in the CF adapter via `waitUntil`.*
- **P1** — `packages/runtime/src/server.ts:73-76` — the `RequestTracer` seam is `startSpan(name)` only: no parent, no incoming `traceparent` honored, none emitted to downstream calls, and the `Tracer`'s own parent/child machinery (`packages/observability/src/tracer.ts:90-107`) is unreachable from the request path. Every trace is a single orphan root span — no db/render/queue children, no cross-service stitching. *Recommendation: parse W3C `traceparent` into the root span, expose the request span via the request context so batteries can open children, and emit `traceparent` from the webhook deliverer.*
- **P1** — `packages/mcp/src/tools.ts:267-279`, `packages/mcp/src/server.ts:40-46` — **agent operations are invisible.** No audit log, no spans, no per-tool-call record of who dispatched `delete_content_entry` or drove `handle_request` with what input. For a framework whose strategic bet is "operated by an agent," the operator's actions are the *least* observable thing in the system. *Recommendation: a mandatory audit sink on `dispatch` (tool, input hash, outcome, duration) — this is also the substrate the authz work (Security P1) needs.*
- **P2** — `packages/runtime/src/server.ts:886-894` — the request span carries `keel.request_id` but `requestId` (a UUID) and `traceId` are unrelated identifiers; the join works but only if operators know to use the attribute. Document it, or derive the span's traceId from an incoming id when present.
- **P2** — `packages/cloudflare/src/fetch-handler.ts:94-96` — the default access log is an unstructured `console.log` line on both tiers; there is no structured/JSON option out of the box, and no metrics surface anywhere (no counters, no latency histograms). Traces-only is a defensible 1.0 cut; say it out loud.
- Good: both tiers log one access entry per request with method/path/status/latency/requestId in the same shape (`runtime/src/server.ts:79-95`, `fetch-handler.ts:59-70`); health + readiness endpoints are on by default (`server.ts:148-152`); the OTLP JSON mapping is byte-pinned by tests (`observability/test/observability.test.ts:211-287`) and correctly handles id-width truncation (`otlp.ts:72`).

## Top launch-blockers

1. **The scaffold→run loop is broken three ways** (`create-keel/src/templates.ts:24,34-53`; `cli/src/bin.ts:113-119`): unpublished `@keel/*@latest` deps, no `@keel/cli` dependency for the `keel dev` script, and a guaranteed `keel.sites.ts` crash. The first five minutes of every new user's experience fails. Nothing else in this review matters to someone who can't boot the framework.
2. **Zero spans on real requests** (`cli/src/run.ts:220-224`; grep: no consumers of `@keel/observability`): the observability story is a well-tested library that nothing calls, with no flush seam on either tier even for opt-in users (`fetch-handler.ts:357-360`).
3. **The MCP control plane is unlaunchable** (`mcp/src/server.ts:24` — zero callers; no `keel mcp` in `cli/src/run.ts:129-147`): the headline "agent-operated" bet cannot be started by any supported path — and when launched manually, has no authz or audit trail.
4. **Webhook SSRF guard is redirect-bypassable** (`webhooks/src/webhooks.ts:296-316`): a documented security invariant defeated by a 302 to the cloud metadata endpoint.

## Recommended plan items

1. **Fix the scaffold loop** — add `@keel/cli` to `templates.ts` deps; make `keel dev` tolerate a missing `keel.sites.ts` (empty sites, app-only dispatch); decide the dependency story (publish `0.x` or scaffold tarball/`file:` pins); land a create-keel e2e that installs and boots the output and curls a route. (Unblocks #1; ~1–2 days.)
2. **Close the webhook redirect bypass** — `redirect: "manual"` in the deliverer, 3xx = `WEBHOOK_DELIVERY_FAILED`, widen `FetchLike`; add the signed-timestamp replay defense to `sign`/`verify` while in the file. (~half day, with tests.)
3. **Wire tracing end to end** — env-driven `Tracer` + `OtlpHttpExporter` construction in `keel serve`/`dev`; interval + drain flush on node; `(request, ctx)` arity + `waitUntil` flush in the CF adapter; one `examples/blog` line proving spans land in a local collector. (Resolves the P0 and the flush P1; ~2 days.)
4. **Ship `keel mcp`** — load `keel.app.ts`, build the context (routes, content db when present), `startMcpServer`; add `mode: "read-only" | "operator"` gating the write/dispatch tools and a mandatory audit sink on `dispatch`. (Makes bet #4 real *and* safe; ~2 days.)
5. **Make the API surface reachable** — `keel openapi [--out openapi.json]` over the existing generator; add a per-route internal/exclude filter. Schema extraction from the Zod boundary validators (ADR 0005) is the natural pre-1.0 follow-on.
6. **Propagation + child spans** — `traceparent` in/out, request span on the request context, first child spans from `@keel/db` and the webhook deliverer. (Turns single-dot traces into actual traces; post-wiring, pre-1.0.)
7. **Widen the loop tests** — CLI e2e for `serve`/`dev`/`deploy --release`/`rollback` as spawned processes; one integration test asserting a served request produces a span and an access entry; classify permanent webhook failures so the queue stops retrying blocked URLs.
