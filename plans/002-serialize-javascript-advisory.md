## Why this matters

`@lesto/content-shared` (published, v0.1.7, not `private`) directly depends on
`serialize-javascript` at `^6.0.0`. That range can **never** resolve to the
patched `7.0.5` and sits inside the range flagged by `bun audit` as **high —
RCE via `RegExp.flags` / `Date.prototype.toISOString()`** (GHSA-5c6j-r48x-rmvq),
plus a CPU-DoS (GHSA-qj8w-gfj5-8c6v). The package imports it at runtime to
serialize collected content into a JS module, so every downstream install of
`@lesto/content-shared` (and `@lesto/content-core`, which imports it too) ships
a serializer with a high advisory on the exact code path that serializes
content data. Two independent audit passes reached this finding. The fix is a
transitive-safe bump and re-lock; the v7 call signature is unchanged for this
usage.

## Steps

### Step 1: Raise the direct pins

- In `packages/content-shared/package.json`, change
  `"serialize-javascript": "^6.0.0"` → `"^7.0.5"` and
  `"@types/serialize-javascript": "^5.0.0"` → `"^5.0.4"`. (There is **no** v7 of
  the `@types` package — latest is `5.0.4`, which content-core already uses; do
  not hunt for a v7.)
- In `packages/content-core/package.json`, change
  `"serialize-javascript": "^7.0.2"` → `"^7.0.5"`.

**Verify**: `grep -rn "serialize-javascript" packages/content-shared/package.json packages/content-core/package.json` → both show `^7.0.5`.

### Step 2: Re-lock

**Verify**: `rm -f bun.lock && bun install` → exit 0.

### Step 3: Confirm the advisory clears

Run `bun audit`. A nested vulnerable copy **will** still resolve after Step 1:
`@content-collections/core@0.9.1` pins `serialize-javascript@^6.0.2`
(`bun.lock`), so the direct bump alone leaves a vulnerable nested `6.0.2` — the
root `overrides` is **required**, not conditional. Add it forcing
`serialize-javascript` to `>=7.0.5` (bun honors npm-style `overrides`):
```json
"overrides": { "serialize-javascript": ">=7.0.5" }
```
then re-run Step 2 and re-check. Note: forcing `@content-collections/core` (a
build-time devDep of content-core) onto v7 is a low but real risk — content-core's
`bun run test` (Step 4) is what catches a break inside it.

**Verify**: `bun audit 2>&1 | grep -i "serialize-javascript"` → no high advisory
line (empty, or only informational).

### Step 4: Prove the content serialization still works

**Verify**:
```
cd packages/content-shared && bun run typecheck && bun run test
cd ../content-core && bun run test
```
→ all pass. These suites cover the `sanitize`/`generator` serialization paths.
